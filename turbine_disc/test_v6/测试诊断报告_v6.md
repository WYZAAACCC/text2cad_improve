### 7.2 榫槽 OCP 崩溃

**错误:** `NCollection_Sequence::ChangeValue` — 60 个 3 站 fir-tree 槽的 boolean cut 操作超出几何内核处理能力。

**处理:** 手动标记 `n_rim_slots.required=False, degradation_policy="may_skip_with_warning"`

**根因:** 架构文档 §17 P0#1 — "300 孔 test case 导致 OCP segfault"。60 个复杂榫槽 × ~10 条边/槽 = ~600 条边参与 boolean cut，OCCT 内核无法处理。

**不是本次修复范围。** 所有 6 轮测试中，榫槽 Runtime 从未成功过。这是几何内核层面的已知限制。

### 7.3 Chamfer 降级

1mm 全局倒角在榫槽邻接区域失败（自动 fallback 0.5mm 也失败），安全降级跳过。

## 9. 本轮发现的问题

### 问题 1：前端 forceRoute 在 Spatial 异步流程中丢失 (P2)

**现象:** `setForceRoute('generative_cad_ir')` 在前端设置后，Spatial modal 弹出并异步等待用户回答，Answer 提交后 `doGenerate` 调用时 `_forceRoute` 可能已被重置或模块重新加载。

**影响:** L1 路由走 Primitive 路径（本次测试中 Primitive 路径也失败了，因为 fir-tree 参数超限）

**临时方案:** 在 API 请求体中直接传 `forceRoute: 'generative_cad_ir'`

**建议:** 在 `handleSpatialAnswers` → `doGenerate` 链路上显式传递 `forceRoute` 参数，而非依赖模块级变量。

### 问题 2：榫槽 OCP 崩溃 — 已知 P0 限制 (P0)

**现象:** 60 个 3 站 fir-tree 槽导致 OCP `NCollection_Sequence::ChangeValue`

**根因:** 架构文档 §17 P0#1，已在 6 轮测试中持续出现

**建议:** 
- 短期：减少槽数到 ≤30 或简化 profile 到 1 站
- 长期：实现分段切割策略（架构文档建议）

### 问题 3：A004 outer_dia 边界敏感 (P3)

**现象:** v5 中 outer_dia=500 触发 A004 (`groove outer radius >= profile max - 1mm`)

**LLM 自适应:** v6 中 LLM 根据 Prompt 中 `outer_dia_mm=496` 的显式指导，正确使用了 496mm

**说明:** 这是 Prompt 引导层面的问题，不是 LLM 能力问题。LLM 在收到显式数值后正确使用。

## 10. v5→v6 修复与根因

### 根因

v5 轮廓缺失 Hub 凸起的根本原因是我在改写 Prompt 时**错误地删除了 Hub 规格**。

| Prompt 版本 | Hub 描述 |
|---|---|
| v4 Prompt | `Hub区域：r=130mm, Z=5→115 (hub前侧过渡)` 和 `Hub区域：r=130mm, Z=77.5→115 (hub后侧过渡)` |
| v5 Prompt | ❌ 只有 `Rim外表面：r=250mm, Z=20→100` 和 `中心填充：r=1mm, Z=0→120` |
| **v6 Prompt** | **`r=130 Z=5→20 \| r=250 Z=20→100 \| r=130 Z=100→115`** |

LLM 在每个版本中都**忠实地按照 Prompt 描述生成了轮廓**。v5 中 Hub 缺失不是 LLM 的错误，而是 Prompt 中缺失了 Hub 规格。

### 修复

在 Prompt 中恢复完整的 5 站外轮廓描述：
```
r=1 Z=0→5 | r=130 Z=5→20 | r=250 Z=20→100 | r=130 Z=100→115 | r=1 Z=115→120
```

LLM 立刻正确生成了带 Hub 凸起的完整涡轮盘轮廓。

### 教训

**Prompt 中的规格必须完整。** LLM 不会"自动补全"缺失的几何特征。如果 Prompt 中没写 Hub，LLM 就不会生成 Hub。这是 Text-to-CAD 系统的正确行为——它应该忠实遵循用户的规格而非猜测。
