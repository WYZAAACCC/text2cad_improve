## 4. Spatial 前端交互阶段

### 4.1 LLM 调用

- **模型:** DeepSeek v4-flash
- **输入:** 用户 Prompt（约 700 字符）
- **任务:** 提取 MechanicalObjectGraphDraft（组件角色、空间关系、未知量）

### 4.2 检测到的 3 个 Unknown

| 问题 ID | 问题 | 类型 | 推荐选项 | 用户选择 |
|---|---|---|---|---|
| `q_UNK_001` | 辐板 (web) 轴向厚度未指定 | numeric_value | A: 等厚 8mm | ✅ A |
| `q_UNK_002` | 轮毂 (hub) 径向壁厚未指定 | numeric_value | A: 壁厚 20mm | ✅ A |
| `q_UNK_003` | 榫槽轴向区间未指定 | feature_location | A: Z=25→95 | ✅ A |

### 4.3 Spatial 前端表现分析

**优点:**
- 正确识别了单组件场景（componentCount=1）
- 3 个问题都切中要害——这些参数确实在 Prompt 中没有明确定量
- 推荐选项合理（8mm web 厚度是高压涡轮典型值）

**问题:**
- Spatial 前端把 Prompt 中"外轮廓连续段"的描述误解为多个组件（bore/hub/web/rim），实际上它们是同一个 axisymmetric 零件的不同 radial 区域，不是独立组件
- 这是因为 Spatial 前端的 `MechanicalObjectGraphDraft` 提取逻辑会将有独特几何描述的区域识别为独立 component
- 对于单组件 axisymmetric 零件，Spatial 前端应该在检测到 `len(components) <= 1` 时自动跳过（代码中有此逻辑：`authoring/spatial/pipeline.py`），但实际未触发，说明 LLM 将 hub/web/rim 识别为了多个 component

### 4.4 答案规范化

用户选择全部推荐选项（均为 option A），`continueSpatial` 成功处理：
- 答案规范化后的空间关系被注入到 constraint_graph
- Phase A 求解器验证通过
- 返回 `spatialGraphKey=cg_spatial_b933ae053ce0`

注意：服务器端 `_run_pipeline()` 没有从 spatial session 中提取 `constraint_graph_json`，因为 session key 是 `spatial_b933ae053ce0` 但代码查找的 key 是 `cg_` 前缀的。这导致 **spatial context 并未注入到 L2 prompt 中**。

### 6.2 轮廓分析 — 与第一轮对比

**第一轮 LLM 输出（6 站区域描述风格）:**
```
bore_bottom: r=60,  Z=5→6      ← 区域描述 + bore 内壁共面
bore_wall:   r=60,  Z=6→115
hub_front:   r=130, Z=5→42.5   ← Z 重叠! (bore 6→115 × hub 5→42.5)
hub_rear:    r=130, Z=42.5→77.5
web:         r=210, Z=42.5→77.5 ← Z 重叠! (hub 42.5→77.5 × web 42.5→77.5)
rim:         r=250, Z=20→100   ← Z 重叠!
```
**问题:** bore 内壁共面 | 多对 Z 重叠 | 无 centerline fill

**本轮 LLM 输出（9 站接近正确的连续轮廓）:**
```
start_fill:  r=1,   Z=0→5      ← ✅ centerline fill
hub_lower:   r=130, Z=5→20     ← ✅ 单值 Z
rim_lower:   r=250, Z=20→42    ← ✅ 单值 Z
[jump]:      r=250, Z=42→42    ← ❌ 零宽度! z_rear == z_front
web:         r=210, Z=42→78    ← ✅ 单值 Z
[jump]:      r=250, Z=78→78    ← ❌ 零宽度! z_rear == z_front
rim_upper:   r=250, Z=78→100   ← ✅ 单值 Z
hub_upper:   r=130, Z=100→115  ← ✅ 单值 Z
end_fill:    r=1,   Z=115→120  ← ✅ centerline fill
```
**问题:** 2 个零宽度"跳变"站点 | 无 Z 重叠 | 无 bore 内壁共面

### 6.3 LLM 意图分析

LLM 试图构建一个真正的 Z 单值外轮廓，但遇到了一个真实的几何约束：

```
轮盘截面示意 (R-Z 平面):

R(mm)
250 ┤        ┌──────────────────┐      ← Rim (外表面)
    │        │                  │
210 ┤   ┌────┘                  └────┐  ← Web (内凹)
    │   │                            │
130 ┤───┘                            └── ← Hub
    │
  1 ┤────────────────────────────────── ← 中心线填充
    │
  0 ┼────┬──────┬──────┬──────┬────┬── Z
    0   20     42     78     100  120

LLM 创建零宽度跳变的原因:
  在 Z=42 处，轮廓需要从 Rim(r=250) 跳到 Web(r=210)
  LLM 用 z_front=z_rear=42 的站点表示"瞬时跳变"
```

**本质问题:** 轮盘的 Web 区域（r=210）内凹于 Rim（r=250），所以纯外轮廓（每个 Z 的最大半径）在 Z=20→100 段始终是 r=250。Web 是侧面凹陷，不应出现在 revolve 的外轮廓中。

### 8.2 A001 错误详情

| # | 站点 | z_front | z_rear | label | 说明 |
|---|---|---|---|---|---|
| 1 | 站点 4 | 42.0 | 42.0 | null | web 入口跳变点 |
| 2 | 站点 6 | 78.0 | 78.0 | null | web 出口跳变点 |

### 8.3 为什么 A006 未触发

本轮 LLM 输出的站点 Z 区间为：
```
(0,5), (5,20), (20,42), (42,42), (42,78), (78,78), (78,100), (100,115), (115,120)
```

零宽度站点 `(42,42)` 和 `(78,78)` 由于 `zf < zr` 恒为 false，不会触发 A006 的重叠检测。A006 只检测到了 0 个重叠——说明 LLM 已经学会了避免 Z 区间重叠。

但 `revolve_profile` 的 preflight 要求 `z_rear > z_front`（A001），所以零宽度站点仍被拦截。

### 9.1 修复方案


### 9.3 Runtime 结果

| 属性 | 值 |
|---|---|
| **STEP 文件** | `output.step` (2,535 KB) |
| **Solid 数量** | 1 (正确) |

⚠️ **这个 STEP 是从简化轮廓生成的**（仅包含 hub inner/outer 和 rim 的纯外轮廓），**不是 LLM 原始输出的完整截面**。它缺少：
- Hub→Web→Rim 的详细过渡截面
- Web 区域的内凹特征
- Hub 的详细壁厚变化

但它证明了：**管线的校验→Runtime→STEP 导出链路完整可用**。一旦 LLM 能生成正确的轮廓，就能产出完整的涡轮盘。

## 11. 本轮新发现的问题

### 问题 1：Revolve 轮廓零宽度站点被 A001 拦截（P2）

**位置:** `validation/geometry_preflight.py` → `dialects/axisymmetric/dialect.py` A001 检测

**根因:** LLM 用 `z_front == z_rear` 创建瞬时径向跳变来转换到不同的外轮廓半径。A001 要求 `z_rear > z_front`（严格大于），零宽度站点被拒绝。

**分析:** 零宽度站点在 LLM 的"连续外轮廓"思维模型中是有意义的 —— 它代表了轮廓中半径的突变。但当前的 revolve_profile IR 语义要求每个站点定义一个实际的 Z 区间段。

**影响:** 如果 LLM 无法用零宽度站点表达半径跳变，它需要用极小（但非零）的 Z 区间来表示。最小间隔受限于 LLM 的输出精度和 schema 约束。

**可能的解决方案:**
1. 在 TURBINE_DISC_REFERENCE_EXAMPLE 中展示如何处理半径跳变（用短段代替零宽度）
2. 在 ProfileStation schema description 中明确说明"z_rear 必须严格大于 z_front，如需要表示半径突变请使用尽可能短的段（如 0.5mm）"
3. 或者放宽 A001 约束，允许 z_rear == z_front（此时站点退化为单点，在 revolve 轮廓中表现为垂直段）

### 问题 2：Spatial 前端对单组件 Axisymmetric 零件产生误判（P3）

**位置:** `authoring/spatial/pipeline.py` → `_extract_object_graph()`

**根因:** Spatial 前端的 LLM 将 Prompt 中描述的 bore/hub/web/rim 区域识别为独立的 component，触发了不必要的空间交互。对于 axisymmetric 单组件零件，这些区域是同一个回转体的不同径向区域，不是独立组件。

**影响:** 增加了 2 轮不必要的 LLM 调用（Spatial start + continue），延长了总响应时间。用户需要额外回答 3 个问题。

**可能的解决方案:**
1. 在 Spatial 前端的 system prompt 中增加 axisymmetric 单组件识别规则
2. 或者先跑 L1 路由，如果结果是 axisymmetric 单 dialect，跳过 Spatial 前端

### 问题 3：Spatial Context 未注入 L2 Prompt（P2）

**位置:** `server/main.py:272-286` `_run_pipeline()`

**根因:** 服务器端从 `_spatial_sessions` 查找 constraint_graph 时使用的 key 与 Spatial 前端返回的 `spatialGraphKey` 不匹配。`continueSpatial` 返回 `spatialGraphKey=cg_spatial_b933ae053ce0`，但服务器代码中 `sd = _spatial_sessions.get(spatial_graph_key, {})` 查找的 key 是前端传入的 `spatialGraphKey`。由于 key 不匹配，spatial context 未被提取。

**影响:** L2 prompt 中缺少空间约束上下文（约束图、组件 placement 信息）。对于单组件 axisymmetric 零件影响较小（不需要装配约束），但对于多组件装配会缺失关键信息。

**需要进一步确认:** 检查 `handleSpatialAnswers()` 中 `doGenerate(_pendingText, _graphKey)` 传递的 `_graphKey` 值是否与服务器端保存的 session key 一致。

### 问题 4：轮廓语义冲突 — Web 内凹 vs Rim 外凸（P1）

**位置:** Prompt 设计层

**根因:** 涡轮盘的 Web（r=210mm）内凹于 Rim（r=250mm），但 Prompt 要求"Z 单值外轮廓连续段"。在 Z=42.5→77.5 区间内，如果取外轮廓（最大半径），则轮廓为 r=250（Rim），Web 内凹不可见。如果 LLM 尝试在轮廓中同时表达 Web 和 Rim，则违反 Z 单值约束。

**本质矛盾:** revolve 只能表达回转体的外轮廓（每个 Z 位置的最大半径）。Web 内凹是侧面的减材特征，不应出现在 revolve 轮廓中。正确的建模方式应该是：
1. revolve_profile：纯外轮廓（rim 形式，r=250 全程覆盖 Z=20→100）
2. cut_annular_groove × 2：从前/后侧面切削出 web 凹陷

**影响:** 这是 LLM 零宽度站点的根本原因。LLM 试图在外轮廓中同时表达 rim 和 web，结果产生了跳变。

**解决方案:**
1. **Prompt 层面：** 明确说明"revolve 只描述外轮廓，web 凹陷用 cut_annular_groove 从侧面切削"
2. **IR 层面：** 考虑增加一个 "side_cut" 类型的操作，专门用于从前/后面切削凹陷
3. **Example 层面：** 在 TURBINE_DISC_REFERENCE_EXAMPLE 中展示用 revolve（纯外轮廓）+ groove（web 切削）的分层建模方式

## 13. 改进建议

### 13.1 高优先级（直击零宽度站点问题）

#### A. 更新 TURBINE_DISC_REFERENCE_EXAMPLE

在 `base_packages/axisymmetric/package.py` 中展示分层建模方式：

```python
TURBINE_DISC_REFERENCE_EXAMPLE = {
    "description": (
        "Turbine disk using layered modeling: revolve the outer silhouette, "
        "then cut web recesses from the side faces."
    ),
    "nodes": [
        {"op": "revolve_profile", "profile": [
            # Pure outer silhouette - one radius at each Z
            {"r_mm": 1, "z_front_mm": 0, "z_rear_mm": 20},
            {"r_mm": 250, "z_front_mm": 20, "z_rear_mm": 100},  # Rim covers full outer surface
            {"r_mm": 1, "z_front_mm": 100, "z_rear_mm": 120},
        ]},
        {"op": "cut_annular_groove", "side": "front",
         "inner_dia_mm": 260, "outer_dia_mm": 500, "depth_mm": 35},
        # ^ This creates the web recess between hub (r=130) and rim (r=250)
    ]
}
```

#### B. 在 L2 system prompt 中添加外轮廓建模规则

```
For axisymmetric revolve_profile:
- The profile defines the OUTER SILHOUETTE only (maximum radius at each Z)
- Recesses, cutouts, and side features MUST be created with separate operations
  (cut_annular_groove, cut_center_bore, etc.)
- Each profile_station MUST have z_rear > z_front (no zero-width stations)
- For radial transitions, use short but positive Z segments (≥0.5mm)
- Do NOT try to model side features (web, seal lands) in the revolve profile
```

### 13.2 中优先级

#### C. 考虑放宽 A001 对零宽度站点的约束

如果零宽度站点在 CAD 意义上等同于垂直段（两个不同半径的点在同一个 Z），revolve 的 sort-by-Z 算法可以正确处理。放宽约束后 LLM 可以用零宽度站点表达半径突变，而不用创建人为的微小段。

**风险:** 需要验证 sort-by-Z 算法对零宽度站点的处理是否产生正确的几何体。

#### D. Spatial 前端跳过单组件 Axisymmetric

在 Spatial 前端 LLM extract object graph 之后，如果 component_count ≤ 1，直接返回成功不提问。代码中已有此逻辑（`len(components) <= 1`），但触发条件是 LLM 正确识别了单组件。需要改进 LLM 的识别规则。

#### E. 修复 Spatial Session Key 不匹配

统一服务器端和前端对 spatial session key 的使用方式。确保 `spatialGraphKey` 能正确索引到 `_spatial_sessions` 中的 constraint graph。
