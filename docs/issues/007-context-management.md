## 背景

DeepSeek V4 支持 1M token 上下文窗口，但这也带来了管理挑战：超大上下文意味着更高的 API 成本；超过上下文窗口会导致 `ContextLengthExceededError`；多轮对话中历史消息会不断膨胀；`reasoning_content`（思考过程）也占用上下文。

当前库不做任何上下文管理，用户需要手动截断消息列表。

**设计决策（已确认）**：`SlidingWindowStrategy` 作为默认策略（零额外成本、零延迟），`ContextCompressor` 作为可选升级（需额外 LLM 调用，用 flash 模型做摘要）。两种策略共享统一的触发条件：估算 token 达到 `max_tokens * 0.8` 时自动触发。

### 策略对比

| | SlidingWindow | ContextCompressor |
|---|---|---|
| 实现复杂度 | 低 | 高 |
| 额外成本 | 零 | 每次压缩一次 flash API 调用 |
| 信息保真度 | 旧消息直接丢弃 | 旧消息提炼为摘要 |
| 延迟 | 无 | 增加摘要 LLM 调用延迟 |
| 默认启用 | **是** | 否（按需） |

## 任务

1. 新建 `seekflow/context.py`
2. 实现 `ContextPolicy` 基类，两种策略继承：
   - `SlidingWindowStrategy(max_messages=50, preserve_system=True, preserve_tools=5)` — 默认。保留 system message + 最近 N 条消息 + 回溯保留完整 tool exchange
   - `ContextCompressor(summary_model="deepseek-v4-flash", max_summary_tokens=500, trigger_ratio=0.8)` — 可选。用 flash 模型对早期消息生成摘要，注入到 system message 后面。摘要结果缓存复用（旧消息未变时不重复调用）
3. 统一的 token 预算估算：
   - #14 上线前：`len(str(messages)) / 4` 粗略估算
   - #14 上线后：切换到 tiktoken 精确计算
   - 触发阈值：估算 token 达到 `max_tokens * 0.8`
4. 保留规则（两种策略共用）：
   - system message 完整保留
   - 最近 N 条 tool exchange 保留（回溯到对应 tool_call，保证消息完整性）
5. 配置入口：`ToolRuntime(..., context_policy=SlidingWindowStrategy(max_messages=50))`，`context_policy=None` 保持当前行为（不做管理）

## 验收标准

- [ ] `SlidingWindowStrategy` 正确丢弃超限旧消息，保留 system + 最近 tool exchange
