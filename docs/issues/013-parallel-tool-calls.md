## 背景

DeepSeek 支持在一次 API 响应中返回多个 `tool_calls`。当前 `ToolExecutor` 对多个 tool call 是逐个串行执行的：

```
call tool_1 → wait → call tool_2 → wait → call tool_3
```

当工具调用之间没有数据依赖时（例如同时查询三个城市的天气），串行总延迟 = t1 + t2 + t3，但并行可以实现 max(t1, t2, t3)。

**设计决策（已确认）**：**不做显式的数据依赖检测。信任 LLM 的并行声明**。DeepSeek 的 tool calling 协议已经区分了并行和串行——同一响应中的多个 `tool_calls` 就是 LLM 认为可以并行的；跨轮次的 tool_calls 就是有依赖需串行的。如果 LLM 出错（极少情况），失败的 tool_call 会在下一轮被 runtime 的 repair 机制自动修正。

### 决策理由

LLM 已经在协议层面声明了并行意图：
- **同一响应多个 tool_calls** → LLM 认为无依赖 → 直接并行
- **跨轮次 tool_calls** → LLM 需要先看结果 → 自然串行

无需额外的依赖检测逻辑。实现焦点放在并行基础设施上。

## 任务

1. 修改 `seekflow/tools/executor.py` 中的 `ToolExecutor.execute_batch()`
2. 并行执行策略：
   - 同一响应中的多个 `tool_calls` → 直接并行执行
   - 使用 `asyncio.Semaphore(max_parallel)` 限流
   - `asyncio.gather(return_exceptions=True)` 保证错误隔离
3. 新增 `max_parallel: int = 5` 配置项
4. 同步版使用 `concurrent.futures.ThreadPoolExecutor(max_workers=max_parallel)`
5. async 版（#9 集成后）使用 `asyncio.gather()`
6. 不实现显式数据依赖检测——信任 LLM 的 tool_calls 顺序

### 为什么不需要依赖检测

```
# 例 1 — LLM 声明并行（同一响应 2 个 tool_call）
response.tool_calls = [
    {"name": "get_weather", "arguments": {"city": "北京"}},
    {"name": "get_weather", "arguments": {"city": "上海"}},
]
→ 同时执行 ✓

# 例 2 — LLM 分两轮（有依赖）
# round 1: tool_calls = [{"name": "get_weather", "arguments": {"city": "北京"}}]
# → result: {"temperature": 28}
# round 2: tool_calls = [{"name": "is_hot", "arguments": {"temperature": 28}}]
→ LLM 自己拆成两轮，自然串行 ✓
```
