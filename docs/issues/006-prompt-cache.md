## 背景

DeepSeek API 支持自动 Prompt Cache：当多个请求的 system message + 前缀 ≥1024 token 且完全一致时，后续请求的缓存命中部分按 ¥0.028/M token 计费（全价的 1.6%），节省约 90% 输入成本。缓存命中时 API 返回：

```python
usage = {
    "prompt_tokens": 50000,
    "completion_tokens": 1000,
    "total_tokens": 51000,
    "prompt_tokens_details": {
        "cached_tokens": 48000,  # 命中缓存的 token 数
    }
}
```

当前库完全不感知缓存机制。用户自己保证前缀一致性才能命中缓存，但容易因 system message 中的时间戳、消息列表微小差异、多轮对话中 system message 被意外修改而导致缓存失效。

**设计决策（已确认）**：采用"被动观测 + 主动告警"两层策略。不做自动前缀优化（太侵入），只做可观测性——让用户知道缓存有没有生效、为什么失效。DeepSeek 的缓存是服务端自动行为，库的责任是"不破坏缓存"而非"主动优化缓存布局"。

## 任务

1. 新建 `seekflow/cache.py`
2. **第一层 — 被动观测**：解析 `usage.prompt_tokens_details.cached_tokens`，在 `ChatResponse.usage` 中保留，日志中输出缓存命中率（如 `Cache hit: 48000/50000 (96%)`）
3. **第二层 — 主动告警**：实现 `CacheSentinel`，对可缓存前缀做 hash，每次请求时对比。前缀变化时发出警告：`"Cache prefix changed — cache invalidated. Keep system message stable for best cache performance."`
4. 可缓存前缀定义：从 system message 开始到最后一个 user message 之前的所有消息。提取并 hash 这段内容，变化时告警
5. 在成本追踪（#11 上线后集成）中区分 cached/uncached token 费用
6. 文档化最佳实践：固定 system message、避免前缀动态内容
