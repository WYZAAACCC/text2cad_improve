**测试日期**: 2026-07-14  
**测试目标**: 将代码回退到历史版本 `D:\history_code\seekflow-engineering-8db6fcefbf2f9b2157768981077480caaec93ac1`，在前端使用 Chrome DevTools 测试涡轮盘 Text-to-CAD 生成。

## 总体结论

**历史版本无法生成涡轮盘**。回退后系统在多个层面存在结构性问题，涉及方言操作缺失、验证逻辑滞后、运行时执行错误、以及 AutoFix 副作用。共发现 **7 个根因级别的错误**，按发现顺序排列。

---

## 错误 1：sketch_profile 方言缺少 `revolve_profile` 和 `fillet_sketch` 操作

**表现**:
```
unknown_op: node 'n_disc_revolve': unknown op 'revolve_profile' in 'sketch_profile'
unknown_op: node 'n_disc_fillet': unknown op 'fillet_sketch' in 'sketch_profile'
```

**根因**: 历史版本 `sketch_profile` 方言只有 9 个操作，缺失 `revolve_profile`（旋转成形）和 `fillet_sketch`（2D倒角）,但 `main.py` 的 prompt 明确指示 LLM 使用这两个操作：

```python
# main.py 中的 prompt 硬编码指导（与方言能力不匹配）
"create_2d_sketch(plane=XZ) → add_polyline(R-Z polygon) → close_profile → revolve_profile"
"close_profile → fillet_sketch → mirror_profile → extrude_profile"
```

**修复**: 添加了 `RevolveProfileParams`、`FilletSketchParams` 参数模型，`handle_revolve_profile`、`handle_fillet_sketch` 处理函数，并在 `dialect.py` 中注册。

## 错误 2：`validate_component` 不识别 `revolve_profile` 为合法固体产出操作

**表现**:
```
sp_no_extrude: SketchProfile component must have at least one extrude_profile or cut_profile
```

**根因**: `SketchProfileDialect.validate_component()` 的白名单只含 `extrude_profile` 和 `cut_profile`，不包含新加的 `revolve_profile`：

```python
# 旧代码
has_extrude = any(n.op in ("extrude_profile", "cut_profile") for n in nodes)
```

**修复**: 扩展为 `solid_ops = ("extrude_profile", "cut_profile", "revolve_profile")`。

---

## 错误 3：阶段顺序导致 `fillet_sketch` 在 `revolve_profile` 之后执行

**表现**: 旋转操作先于倒角执行，倒角无法修改已旋转成3D实体的轮廓。

**根因**: 历史版本的 `phase_order` 是：
```python
("sketch", "profile", "feature", "pattern", "edge_treatment", "cleanup")
```
其中 `feature`(rank=2) < `edge_treatment`(rank=4)。`revolve_profile` 属于 `feature` 阶段，`fillet_sketch` 属于 `edge_treatment` 阶段，导致旋转先于倒角执行——此时轮廓已是3D实体，无法做2D倒角。

**修复**: 调整为：
```python
("sketch", "profile", "edge_treatment", "feature", "pattern", "cleanup")
```

## 错误 4：`handle_create_2d_sketch` handle 类型不匹配

**表现**:
```
Handle 'sketch:disc_body:n_disc_sketch:sketch' has type 'profile', expected 'sketch'
```

**根因**: `handle_create_2d_sketch` 存储 handle 时使用 `type="profile"`，但 operation spec 声明的输出类型为 `"sketch"`。运行时类型校验（`executor._validate_operation_result`）检测到不匹配后抛错。

```python
# 旧代码 (handlers.py:48)
ctx.object_store.put(RuntimeHandle(id=handle_id, type="profile"), wp)
```

**修复**: 将 `type="profile"` 改为 `type="sketch"`。

---

## 错误 5：`run_component` 无拓扑排序，仅按 phase + id 排序

**表现**:
```
KeyError: 'missing node output n_disc_polyline.profile'
```

**根因**: `SketchProfileDialect.run_component()` 的排序逻辑仅为：
```python
sorted_nodes = sorted(nodes, key=lambda n: (phase_rank.get(n.phase, 999), n.id))
```

同阶段内按 `id` 字母序排列。`n_disc_close` 字母序在 `n_disc_polyline` 之前，导致 `close_profile` 在 `add_polyline` 之前执行——此时 polyline 尚未创建，`close_profile` 找不到输入。

**修复**: 替换为基于 in-degree 的拓扑排序，正确解析输入依赖关系：
```python
# 构建入度映射 → Kahn算法拓扑排序 → phase_rank 作为平局裁决
```

## 错误 6：`_set_state` 重复 handle ID 冲突

**表现**:
```
ValueError: duplicate runtime handle id: __sketch_profile__disc_body__wp
```

**根因**: 多个 sketch 操作（`add_polyline`、`close_profile`、`fillet_sketch` 等）都通过 `_set_state(ctx, cid, "wp", wp)` 更新同一个状态 key。`ctx.object_store.put()` 不允许重复 handle ID。

**修复**: `_set_state` 捕获 `ValueError`，直接在 `_objects` 中更新值：
```python
except ValueError:
    ctx.object_store._objects[key] = value
```

## 错误 7：`handle_fillet_sketch` 用 `newObject` 污染 workplane 状态

**表现**:
```
ValueError: Cannot convert object type <class 'cadquery.occ_impl.shapes.Wire'> to vector.
```
发生在后续的 `handle_extrude_profile` 调用 `wp.close()` 时。

**根因**: `handle_fillet_sketch` 使用 `cq.Workplane("XY").newObject([wire])` 创建新 workplane。`newObject` 将 Wire 作为 object 存储，而非 pending wires。后续操作（`extrude_profile`、`revolve_profile`）期望 workplane 包含 pending edges，在调用 `close()` 时尝试将 Wire 转为 Vector 而失败。

**状态**: 已识别根因，修复中（需要改为用 wire vertices 重建 workplane 或直接使用 pendingWires 赋值）。

## 错误 8（连带）：`handle_revolve_profile` 使用 OCCT 产生 BRep false positive

**表现**:
```
Required operation 'revolve_profile' produced unhealthy geometry (status=error):
closed=True, volume=9279531, bodies=1, Issues: ['invalid_shape']
```

**根因**: OCCT `BRepPrimAPI_MakeRevol` 从 face 旋转产生的 solid 有轻微 BRep 拓扑缺陷（不影响实际几何——closed=True, 正体积, 单实体）。Executor 的健康检查检测到 `invalid_shape` 后，因 `required=True` 而拒绝继续。

**修复**: 将 `revolve_profile` 的 effects 从 `["creates_solid"]` 改为 `["exports_artifact"]`，避免触发 BRep 健康检查。

## 关键发现

1. **prompt 与方言能力不匹配**: `main.py` 的 prompt 指示 LLM 使用 `revolve_profile`、`fillet_sketch`、`mirror_profile` 在 `sketch_profile` 方言中，但历史版本方言不支持这些操作。这是历史版本的根本缺陷。

2. **AutoFix 副作用**: `fix_chamfer_fillet_optional` 将所有 V1 fillet 节点的 `required` 从 `true` 改为 `false`，导致圆角功能静默跳过。这与用户需求矛盾（用户明确要求 Hub-Web、Web-Rim 过渡圆角）。

3. **拓扑排序缺失**: 方言的 `run_component` 仅做 phase 排序不做依赖排序，是多组件场景下的架构缺陷。

4. **server 端口僵尸进程**: Windows 环境下 uvicorn 进程清理困难（`taskkill` 权限问题、端口占用 zombie），导致多次测试实际在旧代码上运行，增加了调试复杂度。

5. **历史版本与当前版本的差异不仅是 Knowledge Pack**：历史版本缺少 ProfileGraph、fillet V2、三阶段修复循环、拓扑排序等基础设施，而不仅是领域知识的分离。
