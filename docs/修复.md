# SeekFlow Engineering 历史版本回退后“验证通过、Runtime 失败”深度修复文档

## 一、问题定性

本次问题应当定性为：

> **不完整版本回退造成的 Prompt、Dialect、Validation、Canonical IR、Runtime ABI 和 CadQuery 对象表示之间的版本错配。**

历史回退后，系统中同时保留了不同演进阶段的代码：

1. Web 后端和 L2 Prompt 仍要求模型生成 `revolve_profile`、`fillet_sketch` 等新操作；
2. 历史版本的 `sketch_profile` 方言没有完整注册这些操作；
3. 后续为通过验证，逐项补入了参数模型、handler、OperationSpec 和语义白名单；
4. 但执行顺序、Runtime handle 类型、ObjectStore 更新语义、Workplane 状态及几何健康策略没有同步恢复；
5. 因而系统从“静态验证失败”逐步变成“静态验证通过，但 Runtime 连续失败”。

测试记录显示，前七次主要停留在 registry 和 dialect semantics 阶段，第八、九次已经通过完整验证，但分别在 Runtime 对象类型、调度、状态更新和 Wire 处理阶段失败。-

## 二、为什么验证全部通过，Runtime 仍然失败

当前验证管线主要检查：

* Raw IR 能否解析；
* operation 是否注册；
* 参数是否满足 Pydantic 模型；
* node 与 component 引用是否存在；
* 图是否为 DAG；
* 声明的输入输出类型是否匹配；
* node phase 是否等于 OperationSpec phase；
* dialect 语义；
* 部分参数级几何预检查；
* canonicalization 是否成功。

验证管线本身不会执行 CadQuery handler，也不会查看 ObjectStore 中实际保存的 Python 对象。，IR 可以声明：

```json
{
  "outputs": [
    {
      "name": "sketch",
      "type": "sketch"
    }
  ]
}
```

OperationSpec 也声明输出类型为 `sketch`，静态 typecheck 因此通过。

但历史 handler 实际执行时曾保存：

```python
RuntimeHandle(
    id=handle_id,
    type="profile",
)
```

直到统一 executor 在 Runtime 中执行：

```python
stored = ctx.object_store.get_typed(handle_id)
```

才会发现：

```text
声明类型：sketch
实际 handle 类型：profile
```

因此：

> Validation 证明的是名义契约；Runtime 检验的是行为契约和对象载荷契约。

当前缺失的不是更多 JSON 验证，而是一个独立的 **Runtime Contract Gate**。

# 三、完整故障链分析

## 3.1 Prompt 与历史方言能力不匹配

历史 Web Prompt 指示 LLM 使用：

* `revolve_profile`
* `fillet_sketch`
* `mirror_profile`
* 其他较新操作

但回退后的历史 `SketchProfileDialect` 并没有这些完整能力。

因此最初出现：

```text
registry:
fillet_sketch 未注册
revolve_profile 未注册
```

局部增加 OperationSpec 只能让 registry 阶段通过，不能证明：

* handler 签名正确；
* handler 返回值符合 OperationResult ABI；
* handler 存入的 RuntimeHandle 类型正确；
* handler 保存的实际对象能够被下游操作消费；
* operation effect 正确；
* 几何结果有效。

一个新 operation 必须作为原子变更同时更新：

```text
参数模型
→ handler
→ OperationSpec
→ dialect contract
→ semantic validator
→ AutoFix 模板
→ Prompt/Knowledge Pack
→ Runtime 后置条件
→ 测试夹具
```

不能只补其中两个文件。

---

## 3.2 `validate_component()` 白名单没有同步更新

新增 `revolve_profile` 后，历史语义检查仍只认可：

```python
("extrude_profile", "cut_profile")
```

所以出现：

```text
sp_no_extrude:
SketchProfile component must have at least one
extrude_profile or cut_profile
```

该问题的直接修复是把白名单改成：

```python
solid_ops = (
    "extrude_profile",
    "cut_profile",
    "revolve_profile",
)
```

但这只修复了方言语义声明，不代表 revolve handler 已经正确实现。

---

## 3.3 Graph validator 正确，但历史 Runtime 没有按图执行

验证阶段的 graph validator 会确认：

* 引用节点存在；
* component 引用存在；
* 不存在循环依赖。

但历史 `run_component()` 实际采用：

```python
sorted(
    nodes,
    key=lambda n: (
        phase_rank.get(n.phase, 999),
        n.id,
    ),
)
```

也就是：

1. 先按 phase；
2. 同 phase 按 node ID 字母序。

于是：

```text
n_disc_close
```

可能排在：

```text
n_disc_polyline
```

前面，导致：

```text
KeyError:
missing node output n_disc_polyline.profile
```

验证器没有错，因为 IR 中的依赖图是正确的；错的是 Runtime 执行器没有使用这张图。

正确执行器必须使用 Kahn 拓扑排序：

```text
依赖关系决定先后顺序
phase 只用于 ready queue 中的平局排序
node ID 只用于保证确定性
```

当前主线代码已经增加了拓扑排序，但该逻辑仍散落在单方言和混合方言执行路径中，应进一步抽取为统一 planner。-

## 3.4 IR 声明类型与 Runtime Handle 类型不一致

历史 `create_2d_sketch` 的 OperationSpec 声明：

```python
output_types=["sketch"]
```

但 handler 曾写入：

```python
RuntimeHandle(
    id=handle_id,
    type="profile",
)
```

统一 executor 会分别校验：

1. node 声明的 output type；
2. OperationResult 返回的 value type；
3. ObjectStore 中 RuntimeHandle 的实际 type。

因此该异常是合理且必要的防护：

```text
Handle 'sketch:...'
has type 'profile',
expected 'sketch'
```

这说明 Runtime ABI 校验发挥了作用，而不是 Runtime 过于严格。

---

## 3.5 ObjectStore 被同时用作不可变产物仓库和可变状态仓库

`RuntimeObjectStore.put()` 明确禁止重复 handle ID：

```python
if handle.id in self._objects:
    raise ValueError(
        f"duplicate runtime handle id: {handle.id}"
    )
```

这对 node output 是合理的，因为 node output 应当：

* 唯一；
* 不可覆盖；
* 可追踪；
* 可缓存；
* 可审计。

但 `sketch_profile` 又使用固定 key 存储组件内部可变状态：

```text
__sketch_profile__disc_body__wp
__sketch_profile__disc_body__last_point
__sketch_profile__disc_body__closed
```

每个 operation 都更新同一个 `wp`，因此必然与 append-only ObjectStore 冲突。

错误总结中的临时修复是：

```python
except ValueError:
    ctx.object_store._objects[key] = value
```

这不是合格的长期修复，因为它：

* 直接访问私有字段；
* 绕过 RuntimeHandle 类型校验；
* 不更新 `_handles`；
* 绕过缓存和审计；
* 无法为并发和增量重建提供一致语义。

## 正确修复

在 `RuntimeContext` 中单独增加组件可变状态：

```python
@dataclass
class RuntimeContext:
    # 原有字段……

    component_state: dict[
        str,
        dict[str, object],
    ] = field(default_factory=dict)

    def set_component_state(
        self,
        component_id: str,
        key: str,
        value: object,
    ) -> None:
        self.component_state.setdefault(
            component_id,
            {},
        )[key] = value

    def get_component_state(
        self,
        component_id: str,
        key: str,
        default=None,
    ):
        return self.component_state.get(
            component_id,
            {},
        ).get(key, default)
```

之后：

```python
def _set_state(
    ctx,
    component_id: str,
    field: str,
    value,
) -> None:
    ctx.set_component_state(
        component_id,
        field,
        value,
    )
```

ObjectStore 继续保持不可变 node output 仓库，不再承担组件内部可变状态。

# 四、`fillet_sketch` 的真实问题

错误总结已经发现：

```python
cq.Workplane("XY").newObject([wire])
```

会让 Wire 进入普通对象栈，而不是 CadQuery 的 pending wires，导致后续 `close()` 尝试将 Wire 当成 Vector。

但深读代码和 CadQuery API 后，可以确认 handler 实际存在三个独立错误。

## 4.1 `Wire.fillet2D()` 参数类型错误

当前代码生成整数索引：

```python
indices = [vi]

wire = wire.fillet2D(
    radius,
    indices,
)
```

CadQuery 官方接口要求：

```text
Iterable[Vertex]
```

而不是：

```text
Iterable[int]
```

因此必须先从 wire 中选出真实顶点对象。写法：

```python
vertices = list(wire.Vertices())

if vertex_idx is None:
    target_vertices = vertices
else:
    vi = int(vertex_idx)

    if vi < 0 or vi >= len(vertices):
        raise IndexError(
            f"vertex index {vi} out of range"
        )

    target_vertices = [vertices[vi]]

filleted_wire = wire.fillet2D(
    radius,
    target_vertices,
)
```

---

## 4.2 硬编码 `"XY"` 丢失原始草图平面

涡轮盘旋转截面一般位于 `XZ` 平面。

当前代码：

```python
new_wp = cq.Workplane("XY").newObject(
    [wire]
)
```

会把经过圆角的 `XZ` profile 重新包装成 `XY` Workplane。

即使不立即报错，也会破坏：

* 局部坐标系；
* 旋转轴定义；
* 模型方向；
* 原点偏置；
* 后续空间放置。

必须保留原 Workplane 的 Plane：

```python
new_wp = cq.Workplane(
    wp.plane,
    obj=filleted_wire,
)
```

CadQuery 的 Workplane 构造函数允许直接接收 Plane 对象和 Shape。-

## 4.3 `newObject()` 不会自动把 Wire 放入 pending wires

CadQuery 官方文档说明：

```python
newObject(...)
```

只是把 CAD primitive 放到新 Workplane 的对象栈中。

而：

```python
toPending()
```

才会把 wire/edge 添加到：

```text
pendingWires
pendingEdges
```

`extrude()` 和 `revolve()` 会使用 parent chain 中未消费的 pending wires。最小修复是：

```python
new_wp = (
    cq.Workplane(
        wp.plane,
        obj=filleted_wire,
    )
    .toPending()
)
```

---

## 4.4 `fillet_sketch` 最小可用修复代码

```python
def handle_fillet_sketch(node, ctx) -> dict:
    import cadquery as cq

    from seekflow_engineering_tools\
        .generative_cad.runtime.handles \
        import RuntimeHandle

    from seekflow_engineering_tools\
        .generative_cad.runtime.resolve \
        import resolve_input_object

    wp = resolve_input_object(
        node,
        ctx,
        0,
    )

    cid = node.component
    radius = float(
        node.params["radius_mm"]
    )

    vertex_idx = node.params.get(
        "at_vertex_index"
    )

    wires = wp.wires().vals()

    if len(wires) != 1:
        raise RuntimeError(
            "fillet_sketch requires exactly "
            f"one wire, got {len(wires)}"
        )

    wire = wires[0]
    vertices = list(wire.Vertices())

    if len(vertices) < 3:
        raise RuntimeError(
            "fillet_sketch requires at "
            "least three vertices"
        )

    if vertex_idx is None:
        targets = vertices
    else:
        vi = int(vertex_idx)

        if vi < 0 or vi >= len(vertices):
            raise IndexError(
                f"vertex index {vi} out of "
                f"range [0, {len(vertices) - 1}]"
            )

        targets = [vertices[vi]]

    filleted_wire = wire.fillet2D(
        radius,
        targets,
    )

    # 保留原始 Plane，并恢复为 pending wire。
    new_wp = (
        cq.Workplane(
            wp.plane,
            obj=filleted_wire,
        )
        .toPending()
    )

    ctx.set_component_state(
        cid,
        "wp",
        new_wp,
    )

    ctx.set_component_state(
        cid,
        "closed",
        True,
    )

    handle_id = (
        f"profile:{cid}:{node.id}:profile"
    )

    ctx.object_store.put(
        RuntimeHandle(
            id=handle_id,
            type="profile",
            component_id=cid,
            producer_node=node.id,
        ),
        new_wp,
    )

    return {
        "profile": handle_id,
    }
```

这段属于兼容性修复。

更可靠的长期方案不是继续传递带隐式 context 的 Workplane，而是引入明确的：

```python
@dataclass(frozen=True)
class ProfileValue:
    plane: cq.Plane
    wire: cq.Wire
    is_closed: bool
    source_node: str
```

让 profile operation 返回不可变 `ProfileValue`，feature handler 直接消费 wire。

# 五、`close_profile`、`extrude_profile` 和 `cut_profile` 的状态错误

当前代码中存在：

```python
if closed:
    wp = wp.close()
```

它的语义正好反了：

* `closed=True` 表示已经闭合；
* handler 却再次调用 `close()`。

对于普通 pending edges，这可能偶然没有立即失败；但经过 `fillet_sketch` 后，Workplane 栈顶已经可能是 Wire，再次 close 就会触发：

```text
Cannot convert Wire to Vector
```

## 正确原则

feature handler 不应修改上游 profile。

`extrude_profile`、`cut_profile` 和 `revolve_profile` 应只接受已经完成、已经闭合的 profile。

建议改成：

```python
if not profile_is_closed:
    raise RuntimeError(
        f"{node.op} requires a closed profile"
    )
```

随后直接执行 feature，不再调用 `close()`。

## `close_profile` 也需要修复

当前 handler 在没有 start point 时会发出 warning，但仍然执行：

```python
_set_state(
    ctx,
    cid,
    "closed",
    True,
)
```

这会产生“逻辑上 closed、几何上未 closed”的假状态。

必须改成：

```python
if start_point is None:
    raise RuntimeError(
        "close_profile has no valid start point"
    )

closed_wp = wp.close()

# 进一步验证闭合 wire 存在。
wires = closed_wp.wires().vals()

if len(wires) != 1:
    raise RuntimeError(
        "close_profile did not produce "
        "exactly one wire"
    )
```

只有实际闭合成功后，才能设置 `closed=True`。

# 六、`revolve_profile` 的正确修复

错误总结中采用：

```python
effects=["exports_artifact"]
```

来避开几何健康检查。

这属于危险绕过。

统一 executor 只对：

```text
creates_solid
modifies_solid
cuts_material
adds_material
```

执行几何健康检查。将一个实际创建 solid 的 operation 伪装成 `exports_artifact`，会让无效 B-Rep 绕过检查。条件：

```text
closed=True
volume>0
body_count=1
```

并不能证明 B-Rep 有效。

无效拓扑可能在以下阶段继续失败：

* STEP 导出；
* STEP 重新导入；
* Boolean；
* Fillet；
* 网格化；
* FEA 几何清理。

## 推荐修复

优先使用 CadQuery 原生 Workplane revolve：

```python
solid = wp.revolve(
    angleDegrees=angle,
    axisStart=(0.0, 0.0),
    axisEnd=(0.0, 1.0),
    combine=False,
    clean=True,
)
```

CadQuery 文档说明，Workplane `revolve()` 会使用 parent chain 中未旋转的 wires，并且旋转轴坐标相对于当前 Workplane。 `XZ` 工作平面中的 `(r,z)` profile，需要通过测试确认：

```text
局部 x = 半径 r
局部 y = 轴向 z
旋转轴 = 局部 x=0 的 y 轴
```

对应：

```python
axisStart=(0.0, 0.0)
axisEnd=(0.0, 1.0)
```

修复后恢复：

```python
effects=["creates_solid"]
```

并保留以下检查：

* BRepCheck；
* closed solid；
* positive volume；
* body count；
* bbox；
* STEP 导出；
* STEP 重新导入。

不能通过重命名 effect 绕过检查。

# 七、AutoFix 的错误降级策略

当前 AutoFix 会把：

```text
apply_safe_chamfer
apply_safe_fillet
fillet_sketch
```

统一改成：

```json
{
  "required": false,
  "degradation_policy": "may_skip_with_warning"
}
```

其代码假设圆角和倒角都是 decorative finishing operation。轮盘而言，这个假设不成立。

以下圆角可能属于结构特征：

* Hub-Web 过渡圆角；
* Web-Rim 过渡圆角；
* 榫槽根部圆角；
* 轮缘槽底圆角。

它们会影响：

* 几何连续性；
* 应力集中；
* 疲劳寿命；
* 网格质量；
* 用户明确提出的设计目标。

## 建议增加特征角色

```text
decorative
structural_transition
manufacturing
safety_critical
```

对应降级策略：

| 特征角色                  | 默认是否允许降级 |
| --------------------- | -------: |
| decorative            |       允许 |
| manufacturing         |   由知识包决定 |
| structural_transition |      不允许 |
| safety_critical       |       严禁 |

改变 `required` 不属于安全 alias 修复，应归为语义修改，必须：

* 获得明确策略授权；
* 写入 AutoFix 审计；
* 在 UI 中提示；
* 不得静默执行。

# 八、统一调度器修复

当前单 dialect 和 mixed dialect 分别实现自己的拓扑排序。

建议抽取：

```python
plan_component_execution(
    nodes,
    phase_order,
) -> ExecutionPlan
```

ExecutionPlan 至少应包含：

```python
@dataclass(frozen=True)
class ExecutionPlan:
    ordered_node_ids: tuple[str, ...]
    dependency_edges: tuple[
        tuple[str, str],
        ...
    ]
    plan_hash: str
```

调度规则：

1. dependency 永远优先；
2. phase 只用于 ready queue 平局；
3. node ID 只保证确定性；
4. 检测重复依赖；
5. 检测循环；
6. 输出未调度节点和循环路径；
7. Validation 与 Runtime 使用相同计划或相同 plan hash；
8. plan 写入 metadata。

当前 root output 绑定中还存在：

```python
except KeyError:
    pass
```

这种静默吞错必须删除。

required root 没有产生声明输出时，应立即报错：

```python
raise RuntimeError(
    f"required root node {root.id} "
    f"did not produce {output_name}"
)
```

否则前面的真正错误会被隐藏，最后只得到模糊的：

```text
root node did not produce body
```

# 九、OperationSpec 中的其他潜在 Runtime ABI 问题

当前 `add_slot` 仍是 placeholder：

```python
handler=lambda ctx, nid, p: {
    "profile": f"profile:{nid}"
}
```

而统一 executor 调用约定是：

```python
op_spec.handler(
    node,
    ctx,
)
```

一旦实际运行 `add_slot`，就会出现 handler 参数数量错误。

它同样可以通过 registry 和参数验证，却在 Runtime 失败。

因此必须二选一：

1. 实现真正的 `(node, ctx)` handler；
2. 在生产 registry 中删除 `add_slot`。

禁止 placeholder operation 出现在生产 dialect contract 中。

建议启动时对每个 OperationSpec 执行 registry self-test：

* handler signature；
* params model；
* output 数量；
* output type；
* effect 与 output type 一致性；
* 最小 smoke fixture。

当前 profile operation 还大量使用：

```text
modifies_solid
cuts_material
```

但实际输出是 `profile`。

建议扩展 effect taxonomy：

```text
creates_sketch
modifies_profile
closes_profile
creates_solid
modifies_solid
cuts_material
adds_material
exports_artifact
```

避免 profile 操作冒充 solid 修改操作。

# 十、现有测试体系的缺口

## 10.1 `test_revolve_profile_fix.py` 没有测试生产 handler

该测试直接调用：

```python
cq.Workplane("XZ")
    .moveTo(...)
    .lineTo(...)
    .close()
    .revolve(360)
```

生产 handler 却使用：

```text
BRepBuilderAPI_MakeFace
+
BRepPrimAPI_MakeRevol
```

因此测试证明的是：

> CadQuery 原生 revolve 可以工作。

它没有证明：

> SeekFlow 的 `handle_revolve_profile` 可以工作。
