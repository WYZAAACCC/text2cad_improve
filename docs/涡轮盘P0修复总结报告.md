**修复目标**: 解决涡轮盘生成在 structure 验证阶段失败的问题，同时保证所有修复通用，不影响其他零件构建

## 一、问题背景

### 1.1 失败现象

测试 `turbine_disc/test_46aa1c2b_20260707/` 在 structure 验证阶段（14-stage 管线的第 1 个 stage）失败：

```
component 'disc_body' root_node 'n_disc_done' must output body:solid
component 'fir_tree_cutter' root_node 'n_cutter_done' must output body:solid
```

### 1.2 直接原因

LLM 创建了 **hallucinated passthrough 节点**作为"完成标记"：

- `n_disc_done` (op=`close_profile`): 输入 solid，伪造 outputs=`[body:solid]`
- `n_cutter_done` (op=`close_profile`): 输入 solid，伪造 outputs=`[body:solid]`

但 `close_profile` 的 [OperationSpec](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/sketch_profile/dialect.py#L126-L132) 定义 `input_types=["profile"], output_types=["profile"]`，不能接收 solid 输入。

### 1.3 autofix 修复链条断裂

autofix 的 27 个 fix 未能协同修复此问题：

1. `fix_output_names` (step 9): 按 spec 把 `n_disc_done` outputs 从 `[body:solid]` 改为 `[profile:profile]`
2. `fix_unknown_ops` (step 17): `close_profile` 是合法 op，不删除节点（**destructive severity 被跳过**）
3. `fix_root_node` (step 20): 检查 `n_disc_done` 是 required=True → 直接保留（**不检查输出类型！**）

→ structure 验证检查 root_node 必须输出 body:solid → `n_disc_done` outputs 是 profile → **失败**

## 二、修复方案

### 2.1 P0-1: 修复 `_fix_root_node` 验证输出类型

**文件**: [auto_fixer.py:573-611](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L573-L611)

**问题**: `_fix_root_node` 只检查 `required=True`，不检查 root_node 是否真的输出 `body:solid`。

**修复**: 在 `required=True` 检查后增加 `body:solid` 输出验证：

```python
if root_node_obj and root_node_obj.get("required", True):
    # 通用验证：root_node 必须输出 body:solid（与 structure.py 验证规则一致）
    has_body_solid = any(
        o.get("name") == "body" and o.get("type") == "solid"
        for o in root_node_obj.get("outputs", [])
    )
    if has_body_solid:
        continue  # required=True 且输出 body:solid，保留
    # required=True 但不输出 body:solid → fall through 重新选择
```

**通用性**:
- 与 [structure.py:57-60](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/structure.py#L57-L60) 的 root_node 验证规则一致
- 所有合法零件的 root_node 都指向 solid-producing 节点（垫圈、圆盘、板件、装配体等）
- 不包含任何零件类型判断

### 2.2 P0-2: 实施 passthrough 节点检测与清理

**文件**: [auto_fixer.py:691-789](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L691-L789) + [auto_fixer.py:278](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L278)

**问题**: LLM 误用 `close_profile` 作为 passthrough 节点（输入 solid 输出 solid），autofix 没有清理这种非法节点。

**修复**: 新增 `_strip_passthrough_nodes` 函数，在 `fix_output_names` 之前运行（step 8.5）：

```python
def _strip_passthrough_nodes(doc: dict, dialect_registry=None) -> dict:
    """删除 LLM 误用的 passthrough 节点，并重定向下游引用。

    通用检测规则：节点的实际输入类型（来自上游节点 outputs）与 op spec 的
    input_types 不匹配时，视为 passthrough 误用。
    """
    # 1. 构建 (dialect, op, op_version) → spec 映射
    # 2. 构建 node_id → node 映射
    # 3. 对每个节点，检查输入类型是否匹配 spec input_types
    # 4. 不匹配的节点标记为 passthrough
    # 5. 构建 passthrough_id → (input_node, input_output) 重定向映射
    # 6. 重定向下游节点的引用
    # 7. 重定向 root_node 引用
    # 8. 删除 passthrough 节点
```

**关键设计决策**:

1. **基于 spec input_types 判断**，不基于 op 名称黑名单
   - ✅ 正确: `if input_type not in spec.input_types: delete`
   - ❌ 错误: `if op == "close_profile" and input_is_solid: delete`

2. **在 fix_output_names 之前运行**
   - 如果在 fix_output_names 之后运行，passthrough 节点的 outputs 已被修改为 `[profile:profile]`，下游引用的 `body` 输出名不存在，重定向会失败
   - 在 fix_output_names 之前运行，passthrough 节点的 outputs 还是 LLM 原始的 `[body:solid]`，下游引用的 `body` 输出名存在，可以正确重定向

3. **severity="safe_alias"**
   - 删除的是**非法节点**（输入类型不匹配 spec），这些节点无论如何都会导致验证失败
   - 删除并重定向引用是修复行为，不是破坏行为
   - 不影响合法节点（合法节点的输入类型匹配 spec）

**通用性**:
- 适用于所有 LLM 误用 op 的情况（不只是 close_profile）
- 不包含任何零件类型判断
- 与 [typecheck.py:38-44](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/typecheck.py#L38-L44) 的 input_type_mismatch 规则一致

### 2.3 P0-3: L2 prompt 添加禁止 passthrough 节点的规则

**文件**: [prompts.py:156-163](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/skills/prompts.py#L156-L163)

**问题**: L2 system prompt 没有明确禁止 passthrough 节点，LLM 误以为需要在 component 末尾添加"完成标记"节点。

**修复**: 在 `LEVEL2_AUTHORING_SYSTEM_PROMPT` 中新增规则 34 和 35：

```
34. Do NOT create passthrough/marker nodes at the end of a component. The root_node must
    directly point to a solid-producing node (extrude_profile, revolve_profile, extrude_rectangle,
    boolean_cut, boolean_union, cut_*, add_*, apply_safe_*, place_component, circular_pattern_component, etc.).
    Do NOT append close_profile or other profile-modifying nodes after a solid-producing op —
    that creates an invalid passthrough (solid input to a profile-only op).
35. close_profile may ONLY follow profile-producing ops (add_polyline, add_line_segment,
    add_arc_segment, add_circle, add_slot). close_profile MUST NOT follow extrude_profile,
    revolve_profile, or any solid-producing op — close_profile input_types=["profile"] only.
```

**通用性**:
- 将 op spec 的隐含约束（input_types/output_types）显式化为 prompt 规则
- 适用于所有零件
- 不提及任何特定零件名称

## 三、三层防御体系

本次修复建立了针对 LLM passthrough 节点误用的三层防御：

```
Layer 1: Prompt 防御 (P0-3)
  └─ L2 system prompt 规则 34, 35
  └─ 从源头防止 LLM 创建 passthrough 节点

Layer 2: Autofix 防御 (P0-2)
  └─ _strip_passthrough_nodes 检测并清理 passthrough 节点
  └─ 基于 spec input_types 通用判断
  └─ 重定向下游引用到正确的 solid-producing 节点

Layer 3: Root_node 防御 (P0-1)
  └─ _fix_root_node 验证 root_node 输出 body:solid
  └─ 如果 root_node 无效，重新选择正确的 solid-producing 节点
```

**防御协同**:
- 如果 LLM 遵循 Layer 1，不会创建 passthrough 节点 → 无需 Layer 2, 3
- 如果 LLM 不遵循 Layer 1，Layer 2 会删除 passthrough 节点 → Layer 3 不需要触发
- 如果 Layer 2 没有检测到（边界情况），Layer 3 会重新选择 root_node → structure 验证通过
