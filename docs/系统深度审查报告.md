## 一、执行摘要

SeekFlow Engineering Tools 是一个 Text-to-CAD 智能建模平台，核心目标是将自然语言描述转换为可导入工程软件（SolidWorks/NX）的 STEP 文件。系统采用「LLM 编排 + 严格 IR 验证 + 方言化几何运行时」架构，generative_cad 子系统代表了本项目工程化的最高水平。

**核心结论：**

- **generative_cad 子系统（A-）：** 抽象清晰、设计模式恰当、fail-closed 验证严格、ABI 校验完备。这是整个项目工程质量最高的部分，可作为其余部分的参考标杆。
- **几何内核（中等）：** CadQuery + OCP 集成质量较高，但存在 `boolean_intersect` 占位无实现、schema 上限 360 与 OCCT segfault 阈值 300 之间的危险窗口、pattern handler 未复用 `batch_cut` 等问题。
- **LLM 编排（中等偏高）：** 双编排路径并存导致架构一致性风险；`repair_agent.py` 存在安全后门（静默注入 safety=True）；`auto_fixer` 风险分类未强制执行。
- **后端 API（D+）：** 652 行单体 FastAPI 文件包含路径遍历风险、CORS 全开放、12 处静默 except、无并发控制、无持久化、无测试 — 与"zero-trust tool gateway"定位直接矛盾。
- **前端（B-）：** 技术栈现代但存在模块级全局变量反模式、无 SSE/WS（实际是 axios 轮询）、零测试。

**最严重风险（按优先级）：**

1. `repair_agent.py:201-210` 安全后门 — LLM 省略 safety 字段时修复代理静默注入全 True 合规值，绕过 RawSafety.all_true 与 RawConstraints.fail_closed_flags
2. `CutRimSlotPatternParams.count` schema 上限 360 > 已知 segfault 阈值 300，允许 LLM 触发进程崩溃
3. `boolean_intersect` 在 `operation.py:22` 声明但无 handler 实现，LLM 输出该 op 会 KeyError
4. `main.py:643-648` 路径遍历风险 — `task_id`/`filename` 未校验
5. `executor.py:198-199` `except ImportError: pass` — OCCT 不可用时静默跳过几何校验但 STEP 仍生成

## 二、系统定位与架构总览

### 2.1 项目定位

项目自定位为「zero-trust tool gateway + Text-to-CAD 智能建模平台」，目标用户为航空发动机涡轮盘等关键航空零件的设计工程师。系统通过 LLM（DeepSeek）将自然语言转换为结构化 IR（Intermediate Representation），经 14 阶段验证管线后由方言化几何运行时（CadQuery + OCP）生成 STEP 文件，最终通过 import_gate 门控导入 SolidWorks/NX。

### 2.2 三层架构

| 层 | 位置 | 职责 | 健康度 |
|---|---|---|---|
| 前端 app | `app/text-to-cad/src/` | React UI、空间问答、3D viewport | B- |
| 后端 server | `app/text-to-cad/server/main.py` | FastAPI HTTP、L1/L2 LLM 调用、任务调度 | D+ |
| CAD 子系统 | `integrations/.../generative_cad/` | IR 校验、编译器、方言运行时、STEP 导出 | A- |

### 2.3 核心设计思想

1. **双 IR 层：** RawGcadDocument（LLM 输出，extra="forbid"）+ CanonicalGcadDocument（验证后强类型）
2. **方言注册表：** 6 个预定义方言（axisymmetric/sketch_extrude/composition/sketch_profile/loft_sweep/shell_housing），@lru_cache 冻结单例
3. **编译器 Pass：** Protocol 可插拔，FactPropagationPass + PlannerPass
4. **14 阶段验证管线：** 11 RAW_STAGES + canonicalize + 2 CANONICAL_STAGES，fail-closed 短路
5. **两阶段 LLM 编排：** L1 路由（DialectSelectionPlan）+ L2 创建（tool_schema_compiler）
6. **spatial v6 子系统：** 多轮空间问答 + 空间约束图
7. **LLM 修复循环：** governor 控制的 patch-based 修复

### 2.4 代码规模

- generative_cad 子系统：132 个 .py 文件，~15,000+ 行
- 测试文件：tests/ 70+，integrations/engineering_tools/tests/ 130+
- 前端：~3,000 行 TypeScript/TSX
- 后端：652 行单体 main.py

## 三、架构审查

### 3.1 三层架构分层与耦合

**分层结构清晰：** generative_cad 内部按 ir/validation/compiler/dialects/runtime/pipeline 清晰分层，职责单一，内聚度高。

**耦合度中等偏高，存在硬伤：**

- **后端 → generative_cad 路径级 hack：** `server/main.py:15-17` 通过 `sys.path.insert` 注入跨包路径，而非包级 import。`integrations/` 目录一旦移动，后端立即崩溃。
- **后端无状态外置：** `_tasks`/`_spatial_sessions`（main.py:59-61）是进程内存字典，无水平扩展能力，进程重启全部丢失。
- **generative_cad → 兄弟模块直接耦合：** `builder.py:13-17` 直接依赖 `cadquery_backend.builder`、`common.models`、`config` 等，但属同包内合理耦合。
- **SeekFlow 框架依赖方向正确：** generative_cad 通过 `tools.py:7` `from seekflow import tool` 仅使用 `@tool` 装饰器，SeekFlow 顶层不依赖 generative_cad。

### 3.2 双 IR 层设计

**亮点：**

- `ir/raw.py:113` `model_config = ConfigDict(extra="forbid")` — LLM 无法注入未声明字段
- `RawSafety.all_true`（raw.py:104-109）强制 7 个安全标志字段必须 `is True`
- `RawConstraints.fail_closed_flags()`（raw.py:80-90）强制 `require_step_file/sidecar/closed_solid` 必须显式 True
- `RawNode.validate_required_policy`（raw.py:62-66）required 节点必须 `degradation_policy='fail'`
- canonicalize 转换（`validation/canonicalize.py:28-159`）严格解析所有引用，无 implicit fallback

**潜在问题：**

- `RawNode.params: dict[str, Any]`（raw.py:57）和 `CanonicalNode.typed_params: dict[str, Any]`（canonical.py:58）类型擦除到 dict，而非 Pydantic 模型。运行时 fact rules 需通过 `_typed(node, key, default)`（fact_rules.py:55-59）手动 fallback 取值，失去类型安全。
- canonicalize.py:100-101 把 `validate_params` 得到的 Pydantic 对象再 `model_dump()` 回 dict，丢掉了运行时类型校验机会。
- `safety` 和 `constraints` 在两个 IR 中是同一类型共享，所以"双 IR"实际只对 nodes/components/dialects 是双层，safety 和 constraints 是单层。

### 3.3 方言注册表 @lru_cache 冻结设计

**亮点：**

- `dialects/default_registry.py:39-41` `@lru_cache(maxsize=1)` 提供进程级单例，首次调用构建+冻结，后续零成本读取
- `dialects/registry_core.py:37-38` 冻结后 `register()` 抛 RuntimeError，防止运行时篡改
- freeze 前的 governance 检查（default_registry.py:30-33）在启动期捕获命名违规
- 6 个方言覆盖机械零件主要建模范式：旋转体、草图拉伸、2D 草图、扫描放样、抽壳、装配

**硬伤 — 治理列表不一致：**

- `dialects/registry_core.py:11-28` 的 `FORBIDDEN_PART_TOKENS` 有 15 个 token（包含 `bearing`）
- `dialects/governance.py:22-44` 的 `FORBIDDEN_DIALECT_TOKENS` 有 21 个 token（包含 `bearing_seat` 但不含 `bearing`，新增 `engine_block/cylinder_head/crankshaft/camshaft/piston/connecting_rod`）
- 两套禁止列表语义不同，可能导致 governance 检查与 registry 检查结果不一致

**覆盖完整性：**

- 6 个方言共 41 个 op（实测）
- 缺失：钣金折弯、焊接 bead、自由曲面 BRep 直接构造、管路 routing
- `sketch_profile` 的 `add_slot`（sketch_profile/dialect.py:111-117）是 placeholder handler，实际未实现

### 3.4 编译器 Pass 实现质量

**架构亮点：**

- `compiler/pass_manager.py:16-33` `CompilerPass` 是 Protocol，可插拔，扩展只需实现协议 + 在 `build_compiler_module` 注册
- Pass 不能修改 canonical（pass_manager.py:23-28 文档约束）
- 环境变量 `SEEKFLOW_GCAD_ENABLE_MIDDLE_END` 可禁用整个中间端（config.py:13-19）

**致命硬伤 — FACT_RULES 覆盖率仅 19.5%：**

| 方言 | 总 ops | 已覆盖 | 覆盖率 |
|---|---|---|---|
| axisymmetric | 8 | 4 | 50% |
| sketch_extrude | 11 | 1 | 9% |
| composition | 7 | 3 | 43% |
| sketch_profile | 9 | 0 | 0% |
| loft_sweep | 4 | 0 | 0% |
| shell_housing | 2 | 0 | 0% |
| **合计** | **41** | **8** | **19.5%** |

80.5% 的 ops 在 FactPropagationPass 中被 "silently skipped"（fact_propagation.py:130 `return  # No rule registered for this op — skip silently`），导致：

- 下游 DimExpr `ref` 无法解析上游 fact（FactStore 没数据）
- 编译器诊断对大部分操作失明
- PlannerPass 仅能直接从 typed_params 取值，无 fact 辅助

**架构硬伤 — 拓扑排序重复实现 5+ 处：**

- `analysis/fact_propagation.py:185-225` — Kahn + 按 id 排序
- `pipeline/run.py:467-483` — `_run_mixed_dialect_component` 内 Kahn + 按 id 排序
- `dialects/axisymmetric/dialect.py:309-321` — Kahn + 按 `(phase_rank, id)` 排序
- `dialects/sketch_extrude/dialect.py:314-323` — Kahn + 按 `(phase_rank, id)` 排序
- `dialects/composition/dialect.py:185-201` — Kahn + 按 `(phase_rank, id)` 排序
- `dialects/sketch_profile/dialect.py:234-237` — **直接 `sorted` 按 `(phase_rank, id)`，未做拓扑排序** — 这是潜在 bug，如果节点间有跨 phase 的依赖，执行顺序可能错误

**PlannerPass 阈值硬编码：**

- `planning/risk_model.py` 中 `HOLE_PATTERN_BATCH_THRESHOLD=8`、`HOLE_PATTERN_LARGE_THRESHOLD=120`、`MANY_DESTRUCTIVE_OPS_THRESHOLD=32` 全部模块级常量，无 dataclass 配置，扩展需修改源码

### 3.6 SeekFlow 框架与 generative_cad 边界

## 四、几何内核审查

### 4.1 CadQuery + OCP 集成质量

**集成模式：** 「CadQuery 高层 API + OCP 底层直调」混合策略，整体集成质量较高。

**正面发现：**

- `dialects/geometry_utils/boolean_safe.py:48-66` 对 OCCT 构造器做了双层兼容：先尝试 `BRepAlgoAPI_Fuse(a, b)` 直接构造（OCCT 7.6+），失败则回退到空构造器 + `SetArguments/SetTools`，并对 `SetFuzzyValue` 缺失做了 `AttributeError` 吞掉处理。这是正确的版本容忍写法。
- `runtime/cadquery_runtime.py:16-32` STEP 导出优先用 `OCP.STEPControl.STEPControl_Writer` 嵌入 product name，失败回退到 `cq.exporters.export`。
- `dialects/geometry_utils/ocp_loft.py` 直接使用 `BRepOffsetAPI_ThruSections` + `CheckCompatibility(True)`，绕过 CadQuery `.loft()` 的薄包装以处理异构截面（circle→rectangle→circle）。
- `dialects/geometry_utils/ocp_pipe.py` 实现了 BSpline/M polyline/segmented 三策略，并通过 `path_analysis.py` 做方法选择。
- `dialects/geometry_utils/ocp_cylinder.py:69` 使用 `gp_Ax2` 任意方向定位，避免了 CadQuery 默认 XY 平面限制。

**风险点：**

- 多处采用函数内 `import cadquery as cq` 的延迟导入（如 `handlers.py:32`），虽利于无 OCCT 环境下做静态分析，但每次 handler 调用都付一次 import lookup 成本；对大规模 pattern（如 360 槽）有非必要开销。
- `runtime/geometry_runtime.py:12-35` 的 `GeometryRuntime` 是 `Protocol`，但 `cadquery_runtime.py` 的 `CadQueryRuntime` 未显式声明继承，依赖 structural subtyping；`export_step` 签名多了 `part_name` 参数，与 Protocol 不完全一致。

### 4.2 布尔运算实现策略与 OCCT 限制处理

**Union（`composition/handlers.py:248-344`）四层 fallback：**

1. CadQuery `a.union(b)` + solid-count 验证
2. CadQuery `a.fuse(b)`（实质仍是 OCCT `BRepAlgoAPI_Fuse`）
3. `boolean_union_safe`（`boolean_safe.py:77-142`）：fuzzy fuse 三档（`tol`、`5*tol`、`10*tol`）→ heal_shape + fuzzy fuse → compound（可选）
4. Hard fail，记录 `degraded_features` 但**不**降级返回

策略正确，且 `composition/handlers.py:278` 用 `n_result < n_a + n_b` 判定合并成功是合理的近似。

**Cut（`composition/handlers.py:347-376`）两层：**

1. `target.cut(tool)`
2. `target.cut(tool, ctx.tolerance.boolean_fallback_tolerance)`

**Intersect — 严重缺口：**

- `dialects/operation.py:22` 声明了 `"boolean_intersect"` 字符串
- `validation/geometry_preflight.py:45` 在 `boolean_ops` 列表中包含 `"boolean_intersect"`
- **但 `dialects/composition/handlers.py` 中没有 `handle_boolean_intersect` 实现**
- 全局 grep `BRepAlgoAPI_Common` 零命中

**结论：** intersect 是「占位符 op」，无实现。若 LLM 输出 `boolean_intersect` 节点，会在 `dialect.get_op_spec` 处抛 `KeyError`。这是文档与实现不一致的缺口。

**Batch cut（`boolean_batch.py:41-119`）：**

- 硬上限 500 cutters（`:65-69`），但 `MAX_PATTERN_INSTANCES=360`（risk_model.py:25）才是真正 schema 上限，两处常量未对齐。
- 顺序 fallback 的失败阈值是 50%（`:98`），对 360 孔 pattern 意味着允许 180 个孔失败仍返回「成功」——这与 `handle_cut_rim_slot_pattern` 的「全部失败才 fail」（handlers.py:285）策略不一致。
- `:104-108` 的 `warnings.warn` 是 Python 标准库 warnings，而非 `ctx.warnings`，不会被 runner 收集进 metadata，审计链路断裂。

### 4.3 OCCT 内核阈值处理

| 阈值 | 值 | 位置 | 实际强制 |
|---|---|---|---|
| `HOLE_PATTERN_BATCH_THRESHOLD` | 8 | `planning/risk_model.py:18` | 仅 planner 提示，runtime 不强制 |
| `HOLE_PATTERN_LARGE_THRESHOLD` | 120 | `planning/risk_model.py:22` | 仅 planner warning，runtime 不强制 |
| `MAX_PATTERN_INSTANCES` | 360 | `planning/risk_model.py:25` | axisymmetric schema 中 `cut_rim_slot_pattern` 用 `le=360`，`cut_circular_hole_pattern` 用 `le=240`，两处不一致 |
| `batch_cut` 硬上限 | 500 | `boolean_batch.py:65` | 强制 `ValueError` |
| count ≥ 300 segfault | — | 仅文档提及 | **代码中无任何 300 阈值检查** |

**关键风险：**

- `docs/涡轮盘榫槽切割策略鲁棒性优化报告.md:51` 明确记录「count ≥ 300 OCP segfault 已知限制」，但 `CutRimSlotPatternParams.count` 的 schema 上限是 360（`bases/axisymmetric/models.py:143`），允许 LLM 输出 300-360 之间的 count，触发 segfault 风险。schema 上限与已知 segfault 阈值之间有 60 的危险窗口。
- `handle_cut_circular_hole_pattern`（`handlers.py:130-190`）和 `handle_cut_rim_slot_pattern`（`handlers.py:229-305`）均**未**在 handler 层做 `count >= 300` 的硬阻断，只依赖 schema。一旦 schema 被绕过（如 `typed_params` 直接注入），segfault 会直接 crash 进程而非抛 Python 异常。

### 4.4 拓扑稳定性：union-then-cut vs sequential cut 策略选择

**实现位置：**

- `handle_cut_circular_hole_pattern`（`handlers.py:166`）：`if is_narrow_body or count > 6` → sequential cut
- `handle_cut_rim_slot_pattern`（`handlers.py:259-295`）：`if count <= 6` → union-then-cut；`else` → sequential cut

**策略分析：**

- 阈值 `6` 是硬编码的「magic number」，与 `HOLE_PATTERN_BATCH_THRESHOLD=8` 不一致，且无注释说明为何选 6。
- `handlers.py:262` 注释「6 个槽的 union 约 48 条边，远低于 OCCT 崩溃阈值」——这个估算假设每槽 8 条边，但 fir-tree 截面站点数可变（`SlotProfileStation` min_length=2），实际边数可能翻倍。
- **策略不一致：** `cut_circular_hole_pattern` 在 `count > 6` 时走 sequential，而 `cut_rim_slot_pattern` 在 `count > 6` 时也走 sequential，但 `batch_cut`（`boolean_batch.py`）明确说 batch compound cut 比 sequential 快 10-100x 且更稳定。两个 handler 都没有调用 `batch_cut`，而是各自实现了 sequential 循环。这是明显的代码重复且未利用已有的 `batch_cut` 工具。
- `handle_cut_circular_hole_pattern` 的 sequential 路径（`handlers.py:176-180`）用 bare `except Exception` 吞掉单孔失败并 `ctx.warnings.append`，**没有**记录到 `ctx.degraded_features`，丢失了审计痕迹；而 `handle_cut_rim_slot_pattern`（`handlers.py:296-299`）正确记录了 `degraded_features`。两处同类操作处理方式不一致。

### 4.5 几何错误恢复机制

**核心机制：** `runtime/recovery.py:22-110` 的 `handle_feature_failure`：

- required=True → `RuntimeError`（hard fail）
- required=False + `degradation_policy == "may_skip_with_warning"` → warn + record + 返回原 body
- 其他组合 → hard fail

**正面：** 统一入口，fail-closed 默认，审计字段完整（`warnings`、`degraded_features`、`operation_metrics` 三处记录）。

**缺陷：**

- `handle_apply_safe_chamfer`（`handlers.py:312-345`）**没有**调用 `handle_feature_failure`，而是自己实现了等价逻辑（distance → distance/2 → degrade），导致：
  - `:333-337` 手动写 `ctx.operation_metrics.append({"status": "degraded"})`，与 `handle_feature_failure` 的 `:83-88` 重复实现
  - `:340-344` 在非 may_skip 场景调用 `handle_feature_failure`，但前面 `:319-339` 的 may_skip 路径已经返回，逻辑分支冗余
- `handle_cut_circular_hole_pattern` 的单孔失败（`handlers.py:176-180`）用 bare `except` + `ctx.warnings`，**完全绕过** `handle_feature_failure`，无 `degraded_features` 记录。
- `runtime/recovery.py:62` 的 `getattr(node, "required", True)` 默认 True 是安全的 fail-closed，但 `getattr(node, "degradation_policy", "fail")` 默认 "fail"（`:72`）意味着未显式声明 degradation_policy 的 required=False 节点会 hard fail — 这可能不是 LLM 输出的默认预期。

### 4.6 核心 handler 实现质量

#### `handle_revolve_profile`（`handlers.py:31-105`）

- **Z 重叠检测**（`:60-73`）：O(n²) 双重循环检测 station 区间重叠，正确拦截「bore/hub/web/rim 分区域描述」模式。
- **多 station 路径**（`:75-96`）：`pts_2d.sort(key=lambda p: p[1])` 只按 z 排序，`:82-85` 的 dedup 用 `pt != unique_pts[-1]` 严格相等比较浮点数，对 `z_front_mm` 精度不一致的输入可能漏 dedup。
- **缺失检查：** 未验证 `r_mm` 单调性（profile 可能自相交），未验证 profile 是否触及旋转轴（r=0 处可能产生退化几何）。

#### `handle_cut_circular_hole_pattern`（`handlers.py:130-190`）

- **死代码**（`:150-163`）：`try: from OCP.BRepExtrema import ...` 块导入了 `BRepExtrema_DistShapeShape`、`gp_Pnt`、`gp_Ax1`、`gp_Dir`、`gp_Circ`、`GC_MakeCircle`、`BRepBuilderAPI_MakeEdge`、`BRepBuilderAPI_MakeWire`，但**全部未使用**，只计算了 `pcd_radius`（未使用）和 `is_narrow_body`（用 bbox 比较，未用任何 OCP API）。这是残留的调试代码或未完成的实现。
- **`is_narrow_body` 判断**（`:161`）使用 `bb.xlen < pcd + hole_dia`，对圆形旋转体 `bb.xlen == 2 * max_r`，这个判断等价于 `max_r < pcd/2 + hole_dia/2`，即孔超出外径 — 但这种情况 preflight 已拦截，所以 `is_narrow_body` 实际只在非旋转体零件上生效，而该 dialect 就是 axisymmetric，此分支几乎永远为 False。

#### `handle_cut_rim_slot_pattern`（`handlers.py:229-305`）

- **slot 截面构造**（`:247-257`）：构造的是半个 fir-tree 截面（对称），但 `SlotProfileStation` 只要求 `depth_mm` 非递减（`models.py:131-135`），未约束 `half_width_mm` 单调性，可能产生自相交截面。
- **`outer_r` 估算**（`:246`）：`max(bb.xlen, bb.ylen) / 2.0`，对非圆形旋转体（如带槽的）会高估 outer_r，导致槽 cutter 不贴合实际外缘。

#### `handle_apply_safe_chamfer`（`handlers.py:312-345`）

- 见 §4.5 分析，未复用 `handle_feature_failure`。
- `_chamfer_by_target`（`:350-361`）的 `target == "all_external_edges"` 调用 `body.chamfer(distance)`，但 CadQuery 的 `Workplane.chamfer` 要求至少选中一条边，对空 Workplane 会抛异常 — 这里没有先选中 edges，依赖 CadQuery 内部默认行为，脆弱。
- `:358-360` 的 `for e in edges: try: result = result.edges(e.edge_index).chamfer(distance) except: continue` 会静默吞掉所有边失败，最终返回的 `result` 可能与输入相同，但调用方无法区分「无 chamfer 需求」和「所有边 chamfer 失败」。

### 4.8 STEP/STL 导出正确性

**STEP 导出**（`runtime/cadquery_runtime.py:16-32`）：

- 优先 `STEPControl_Writer` + `STEPControl_AsIs` 模式，调用 `writer.Transfer(solid_obj.wrapped, STEPControl_AsIs)`。
- `:25` 检查 `hasattr(writer, 'SetName')` 后设置 product name，兼容旧 OCCT。
- fallback 到 `cq.exporters.export(solid_obj, str(out_step))`。
- **风险：** `solid_obj.wrapped` 假设输入是 CadQuery Workplane/Solid；若传入 `cq.Assembly`（composition 可能产生），`.wrapped` 不存在，会抛 `AttributeError` 进入 fallback，fallback 的 `cq.exporters.export` 对 Assembly 会调用 `exportAssembly` 而非单 STEP 文件，产物可能不符合预期。`run.py:547-549` 的 `_export_final_solid` 直接传 `ctx.object_store.get(handle_id)`，未做类型归一化。

**STEP 后验证**（`runtime/geometry_postcheck.py:143-187`）：

- 文件存在性、大小（`min_size_bytes=200`，`run.py:287` 传入）、`ISO-10303-21` header 检查。
- **缺陷：** 未验证 STEP 文件中实体数（`ENTITY` 行计数）、未调用 OCCT 重新读入 STEP 做闭环验证。`min_size_bytes=200` 过低，一个空 HEADER 的 STEP 文件约 200 字节，可能通过检查。

**STL 导出：**

- 全局搜索 `exportStl|export_stl|STLExport` 在 `generative_cad` 子系统内**零命中**。
- `pipeline/run.py` 只调用 `ctx.geometry_runtime.export_step`（`:548`），无 STL 路径。
- 若需 STL，需调用方自行用 `cq.exporters.export(solid, out, exportType="STL")`，但 runner 不提供该入口。
- **前端 `Viewport3D.tsx:33` 使用 `useLoader(STLLoader, model.stlUrl || '')` 加载 STL**，但后端无 STL 导出能力 — 前后端契约不一致，STL 模型来源不明（可能是预置数据集）。

**A001（zero-width station，dialects/axisymmetric/dialect.py:118-144）：**

1. **repair_agent.py:201-210 安全后门：** LLM 省略 safety/constraints 字段时，修复代理静默注入全 True 合规值，绕过 RawSafety.all_true 和 RawConstraints.fail_closed_flags

8. **_fix_chamfer_fillet_optional 弱化契约**（auto_fixer.py:1024-1039）：所有 chamfer/fillet 被标记为 optional，runtime 可静默跳过
9. **hole_semantics 阶段命名混淆**（hole_semantics.py:28）：报告为 "geometry_preflight" 阶段，可能影响下游 metadata 解析和 import_gate 判断

## 十、性能与可扩展性评估

### 10.1 LLM 调用并发与限流

| 问题 | 位置 | 风险 |
|---|---|---|
| 无并发限制 | `main.py:433` 每请求新 daemon 线程 | 高并发下线程爆炸、API 配额耗尽 |

| 无全局限流器 | 全局 | 多用户并发时无保护 |

### 10.2 几何运算资源管理

| 问题 | 位置 | 风险 |
|---|---|---|
| GIL 限制 | `main.py:433` threading | cadquery CPU 密集，实际串行 |
| 无内存限制 | `runtime/context.py` ObjectStore | 大模型可能 OOM |
| 无超时 | `_run_pipeline` 无整体超时 | LLM/几何运算卡死时任务永久 processing |

**优美之处：** `runtime/cache.py` 存在增量重建缓存机制（`executor.py:42-55`），避免重复计算。

### 10.3 后端异步任务队列

- **无队列：** `threading.Thread` 直接启动，无 Redis/RQ/Celery
- **无持久化：** 内存字典 `_tasks`/`_spatial_sessions`
- **无任务恢复：** 进程重启后所有进行中任务丢失，客户端轮询 404
- **无水平扩展：** 内存状态绑定单进程，无法多实例部署

### 10.4 性能瓶颈总结

| 瓶颈 | 严重度 | 影响 |
|---|---|---|
| GIL 限制几何运算串行 | 高 | 多用户并发时性能崩溃 |
| 无任务队列 | 高 | 无法水平扩展 |
| LLM client 每次新建 | 中 | 高频调用下 TCP 开销 |
| pattern handler 未复用 batch_cut | 中 | count=360 时性能差 10-100x |
| 前端无 LOD | 中 | 大 STL 渲染崩溃 |
| OperationCache 无上限 | 中 | 长会话内存泄漏 |
| 轮询 5 秒间隔 | 低 | 用户感知延迟 |

## 十四、结论

### 14.1 系统能否实现其想要实现的功能？

**部分实现。** Text-to-CAD 主链路（自然语言 → LLM → IR → 验证 → 几何运行时 → STEP → import_gate → 工程软件）已完整实现，generative_cad 子系统的工程化水平（A-）足以支撑核心功能。但存在以下功能缺口：

- `boolean_intersect` 占位无实现
- `apply_safe_fillet` 定义但未注册
- STL 导出未实现（但前端期望 STL）
- `generative_cad_build_with_autofix` 和 `generative_cad_full_authoring` 工具注册遗漏

### 14.2 系统的性能如何？

**中等偏下。** 主要瓶颈：

- GIL 限制导致几何运算实际串行
- 无任务队列，无法水平扩展
- LLM client 每次新建，连接池无法复用
- pattern handler 未复用 batch_cut，count=360 时性能差 10-100x
- 前端无 LOD，大 STL 渲染崩溃
- OperationCache 无上限，长会话内存泄漏

### 14.3 系统的鲁棒性如何？

**中等。** 14 阶段验证管线的 fail-closed 设计严格，但存在多个绕过漏洞：

- `repair_agent.py` 安全后门允许 LLM 绕过 safety 验证
- `executor.py:198-199` ImportError fail-open
- `auto_fixer` 风险分类未强制执行
- OCCT segfault 阈值与 schema 上限之间的危险窗口
- `import_gate` 信任 metadata 自报告

### 14.4 几何内核的性能与鲁棒性如何？

**性能中下，鲁棒性中等。**

- 性能：CadQuery+OCP 集成质量高，但 pattern handler 未复用 batch_cut，sequential cut 在 count=360 时性能差
- 鲁棒性：错误恢复机制设计良好（`handle_feature_failure`），但多个 handler 绕过它（chamfer、单孔失败）；OCCT 版本兼容处理到位；但 schema 上限与 segfault 阈值之间有危险窗口

### 14.5 代码是否足够健康、强壮与优美？

**核心模块优美，外围模块欠工程化。**

- **generative_cad 子系统（A-）：** 抽象清晰、设计模式恰当、严格 ABI 校验、fail-closed、注释解释 why、测试充分 — 是整个项目工程质量最高的部分，可作为其余部分的参考标杆
- **main.py（D+）：** 652 行单体文件，路径遍历风险、CORS 全开放、12 处静默 except、无并发控制、无持久化、无测试 — 与核心模块的工程化水平严重不匹配
- **前端（B-）：** 技术栈现代，但模块级全局变量反模式、无 SSE/WS、零测试

### 14.6 最终评价

SeekFlow Engineering Tools 是一个**核心模块工程化水平高、但外围包裹层欠工程化**的系统。generative_cad 子系统的双 IR 设计、方言注册表、编译器 Pass、14 阶段验证管线代表了工业级 CAD 生成系统的优秀架构实践。然而，这个精心设计的核心被一个 652 行单体 FastAPI 文件包裹，后者包含路径遍历、CORS 全开放、静默 except、无并发控制等多重生产风险。

**核心矛盾：** 项目自定位为"zero-trust tool gateway"，但 `main.py` 的实现与这一原则直接矛盾。generative_cad 子系统的 fail-closed 设计被 `repair_agent.py` 的安全后门和 `executor.py` 的 ImportError fail-open 削弱。

**改进路径：** 应优先处理 P0 级风险（repair_agent 安全后门、schema 上限对齐、路径遍历、ImportError fail-open），然后处理 P1 级风险（boolean_intersect、工具注册、死代码、CORS、错误处理），最后规划 P2 级改进（FACT_RULES 覆盖、拓扑排序统一、编排路径统一、任务队列、测试补齐）。
