# Knowledge Pack 与三级修复循环最终问题及修复报告

## 一、最终架构判断

你的判断是正确的：

> **不能因为一个枞树形榫槽，就为它单独开发一套完整的参数化求解器；否则遇到异形孔、非规则加强筋、复合流道、异形减重腔、复杂叶根、异形凸台时，又要逐个开发专用模型，Generative CAD 最终会退化成传统参数化零件库。**

因此，系统的最终技术路线应当明确为：

> **通用 CAD 操作语言 + 专业 Knowledge Pack Prompt + 通用几何验证 + 多层 Repair Loop。**

Knowledge Pack 不是“参数化模型”，也不是“榫槽生成函数”，而应当是：

> **约束 LLM 工程推理、几何规划和修复行为的专业知识策略包。**

LLM仍应保留以下自由：

* 决定局部拓扑；
* 选择直线、圆弧、样条、拉伸、扫掠、布尔等构造方法；
* 决定特征分解顺序；
* 根据 OCC 错误替换局部构造路线；
* 针对不同用户要求形成不同榫槽或异形结构；
* 在不改变工程意图的情况下修复局部几何。

底层只需要提供通用能力，例如：

```text
线线求交
线圆求交
圆弧构造
切向检查
镜像
偏置
轮廓闭合
自交检测
最小边长检查
最小壁厚检查
B-Rep合法性检查
布尔结果检查
```

这些能力可服务于所有异形结构，不属于“为每个榫槽单独参数化建模”。

# 二、最重要的新发现：公开 `main` 入口并没有接通你描述的完整系统

截至本次复核的公开 `main` 分支，Web 生产入口实际执行的是：

```text
L1
→ L2
→ 14阶段Validation
→ AutoFix一次
→ 再Validation一次
→ Runtime一次
→ 完成
```

当前 `app/text-to-cad/server/main.py` 中没有接入：

* Knowledge Pack Registry；
* `compile_l1_summary()`；
* `compile_l2_knowledge()`；
* 5轮 LLM Validation Repair；
* Repair Governor；
* L2 Retry；
* 3轮 Runtime Repair。

验证失败时，当前入口只执行一次 `auto_fix_with_report()`；再次验证仍失败便立即结束。Runtime 也只执行一次。([GitHub][1])

这意味着：

> 你设计的 Knowledge Pack 和三级修复模块虽然存在于仓库中，但当前 Web 主入口没有把它们组成你所描述的最终架构。

这是目前最高优先级的问题，不是榫槽知识是否足够详细，而是**主链路尚未接通**。

# 四、Knowledge Pack 当前没有真正进入 L1 和 L2

## 4.1 L1 中的 Domain Skill 实际不可见

`build_level1_routing_prompt()` 会加载 `generic_mechanical` 和 `turbomachinery_reference`，但只是把它们放进返回字典的 `domain_skills` 字段，没有拼接进实际发送给模型的 `user_message`。

Web 主入口调用 LLM 时，只传：

```python
l1["system"]
l1["user"]
```

没有传 `l1["domain_skills"]`。

因此 L1 路由模型实际上只看到了用户请求、Primitive Catalog 和 Dialect Catalog，没有看到专业 Knowledge Pack 摘要。([GitHub][2])

## 4.2 L2 也没有消费 `selected_domain_skills`

L1 Schema 中存在 `selected_domain_skills`，主入口也会为缺失版本补默认值，但之后没有加载这些 Pack。

`build_level2_authoring_prompt()` 当前自动加载的是各 Dialect 的 BasePackage 使用说明，而不是 `knowledge/packs` 目录中的 Knowledge Pack。([GitHub][2])

因此当前代码中的：

```text
BasePackage
```

和：

```text
Knowledge Pack
```

是两条未打通的系统：

* BasePackage 告诉 LLM 某个操作怎样调用；
* Knowledge Pack 告诉 LLM专业结构应怎样设计；
* 现在只有前者进入了 L2。

# 五、全局硬编码榫槽规则必须移除

当前 `main.py` 无论用户是否请求榫槽，都会向 L2 添加：

```text
For fir-tree slot cutters:
sketch_profile
→ create_2d_sketch
→ add_polyline
→ close_profile
→ fillet_sketch
→ mirror_profile
→ extrude_profile
```

这会产生两个问题。

第一，**专业知识污染**。用户生成支架、壳体、叶轮或普通盘件时，也会收到榫槽指令。

第二，**把单一构造顺序写死**。这恰恰会让系统退化成隐性的参数化模板，限制 LLM 根据不同几何选择：

* 显式圆弧；
* 三点圆弧；
* 镜像开放轮廓；
* 完整轮廓；
* 局部圆角；
* 分段布尔；
* 扫掠刀具体。

而且该全局序列中的“先 close、再 fillet、再 mirror”与当前榫槽 Knowledge Pack 中的“先生成单侧、再 mirror、最后 close”也不一致。([GitHub][1])

正确方式是：

```text
不在 main.py 硬编码任何具体零件知识

由L1选择Pack
→ Resolver加载Pack
→ Prompt Compiler动态注入
```

# 六、Knowledge Pack 的正确定位

Knowledge Pack 不应描述一组固定坐标，也不应要求系统调用一个榫槽专用函数。它应该包含六类内容。

## 6.1 专业语义

例如：

* M_A1 是楔角；
* M_A2 是齿形角；
* M_A3 是压力角；
* M_B1～M_B4 是不同工程角色的圆弧；
* LM_WW 是非工作面配合间隙；
* MM_A 是榫槽中心平面与盘子午面的夹角。

上传文档确实将榫槽建模定义为拓扑约束、尺寸约束、中心线对称约束及部件间关联约束的组合，并给出了上述关键参数体系。

## 6.2 不变量

例如：

```text
轮廓必须闭合
不得自交
需要对称时必须在容差内对称
设计圆弧必须具有对应半径
相邻线弧必须连续
工作面与非工作面不得交换
阵列数量必须等于MM_N
布尔切除后的槽数不能减少
```

这些不变量可以通过通用验证器检查，不需要开发榫槽专用参数化模型。

## 6.3 构造策略候选

Knowledge Pack 不应只有一条固定 Construction Sequence，而应提供多个允许方案：

```yaml
strategies:
  - id: half_profile_mirror
    suitable_when:
      - symmetric_profile
    operations:
      - semantic_segments
      - mirror
      - close

  - id: full_profile_explicit
    suitable_when:
      - asymmetric_profile
      - mixed_tooth_geometry

  - id: polyline_then_selected_fillet
    suitable_when:
      - source_geometry_defined_by_lines_and_fillet
```

由 LLM 根据用户需求选择，而不是由 Pack 强制所有模型使用同一套坐标和操作顺序。

## 6.4 禁止性规则

例如：

* 不得将四种圆弧无理由合并成一个参数；
* 不得对所有内部顶点统一倒圆；
* 不得把工作面间隙误施加到非工作面；
* 不得忽略 MM_A；
* 不得在 OCC 失败后删除 required 特征。

## 6.5 自检问题

不是让系统直接判断“必须有24个点”或“绝不能有24个点”，而是给 Repair Agent 提供检查方向：

```text
是否采用了不带语义的固定点模板？
是否保持了工作面/非工作面身份？
是否所有设计圆弧都可追踪到对应参数？
是否存在重复坐标或极短边？
```

## 6.6 修复建议

Knowledge Pack 应额外包含：

```yaml
repair_guidance:
  arc_failure:
    - verify start/end radius consistency
    - try exact center-defined arc
    - try three-point arc
    - reverse arc orientation
  mirror_failure:
    - verify selected axis against coordinate convention
    - ensure half-profile endpoints lie on symmetry axis
  boolean_failure:
    - extend cutter beyond target thickness
    - fuse cutter copies before cut
    - split pattern into batches
```

这类知识能保留 LLM 的自由性，同时提升 Repair Loop 的成功率。

# 七、榫槽 Knowledge Pack 的具体问题和修订意见

## 7.1 坐标方向自相矛盾

当前 Pack 写道：

```text
X = radial inward from rim
slot depth direction = negative X toward disc center
```

如果正 X 已经是由轮缘指向盘心，则槽深方向应是正 X；如果盘心方向为负 X，X 就应定义为径向向外。两句话不能同时成立。([GitHub][3])

建议统一为：

```text
原点：槽口中心
+X：从轮缘向盘心
+Y：切向
中心线：Y=0
槽内深度：X≥0
```

这只是统一坐标契约，不是参数化建模。

## 7.2 镜像轴写错

Pack 定义中心线为 `Y=0`，单侧轮廓为 `Y≥0`。

把 `Y≥0` 映射到 `Y≤0`，应关于 **X轴** 镜像，而不是关于 Y 轴镜像：

```text
mirror_profile(axis="X")
```

当前 Pack 写的是：

```text
mirror_profile(axis="Y")
```

会改变 X 坐标，而非正确复制切向另一侧。([GitHub][4])

## 7.3 M_R1 不能直接当作槽口阵列半径

文档定义 M_R1 为第一榫齿节线基准面距盘轴线的径向距离，而当前 Pack 的局部原点设置在槽口中心，又直接使用：

```text
radius_mm=M_R1
```

这会把“节线半径”当成“槽口原点半径”。([GitHub][3])

这里不需要开发专用求解器，只需让 Prompt 明确：

```text
M_R1是基准节线半径；
槽刀具的阵列定位半径应根据局部基准位置进行换算，
不得未经推导直接等同于槽口半径。
```

LLM负责生成换算关系，Validation检查结果是否与局部坐标定义一致。

## 7.4 MM_A 完全缺失

当前 Construction Strategy 中没有出现 MM_A，只执行垂直于草图的双向拉伸。([GitHub][3])

Prompt 应要求：

```text
如果MM_A非零：
刀具体或其拉伸方向必须体现MM_A；
不得仍使用未旋转的直槽刀具。
```

Runtime 后再通过通用方向向量检查确认槽轴线与规定角度一致。

## 7.5 “固定24点”不应是 Hard Rule

当前 Self-check 把恰好24点的双侧折线视为 Hard 错误。但24点本身并不一定错误；错误的是：

> 在没有专业语义和几何验证的情况下机械套用固定24点模板。

因此应把规则从 `hard` 降为 `heuristic`：

```yaml
severity: heuristic
action: request_additional_validation
```

([GitHub][5])

## 7.6 四个半径不能“隐式共用”，但数值可以相等

当前规则一方面允许用户明确给出相等值，另一方面 Self-check 又规定四个半径不得全部相等，存在逻辑冲突。([GitHub][4])

正确规则应是：

```text
M_B1～M_B4必须是四个独立语义参数；
不得因为简化而自动绑定为同一个变量；
但用户显式给出的数值可以相等。
```

验证参数身份，而不是禁止数值相等。

## 7.7 不应强制 LLM 手算后直接相信坐标

Pack 当前写道：

> LLM calculates control points, tangency points, and arc centers.

该思想可以保留，但必须改成：

```text
LLM提出控制点、切点和圆心
→ 通用Geometry Preflight验证
→ 失败后将具体数值误差交给Repair Agent
```

而不是：

```text
LLM给出坐标
→ 只要JSON合法便直接送入OCC
```

需要增加的不是榫槽专用 Solver，而是通用检查：

```text
|distance(start, center)-R| < tolerance
|distance(end, center)-R| < tolerance
切线夹角误差 < tolerance
轮廓闭合误差 < tolerance
无自交
最小边长 > tolerance
```

所有异形轮廓都能复用这些检查。

# 八、三级修复循环的最终正确架构

建议将系统定义为两个修复域、四种恢复机制。

```text
用户Prompt
  ↓
L1路由
  ├─ Dialect摘要
  └─ Knowledge Pack摘要
  ↓
Knowledge Gate
  ├─ Pack存在
  ├─ 版本有效
  ├─ 状态允许
  ├─ required_dialects满足
  └─ 冲突已解决
  ↓
L2 Author
  ├─ 用户请求
  ├─ 通用Dialect说明
  ├─ Knowledge Pack专业Prompt
  ├─ 反例
  └─ 空间约束
  ↓
14阶段Core Validation
  ↓
Knowledge Compliance Validation
  ↓
① Safe AutoFix
  ↓
② LLM Repair Loop ×5
  ↓
③ L2 Regeneration
  ↓
Runtime
  ↓
④ Runtime Repair ×3
  ↓
STEP / STL / Metadata
```

## 8.1 第一层：Safe AutoFix

AutoFix只能处理：

* 字段别名；
* 版本号格式；
* 缺省空对象；
* 明确可推导的输出名；
* 无歧义的Schema格式问题；
* 数字字符串转数字。

不能自动处理：

* 镜像轮廓；
* 增加齿；
* 删除未知 required 节点；
* 猜测孔位置；
* 猜测圆角半径；
* 修改工作面和非工作面；
* 改变槽型拓扑。

当前 `fix_slot_half_profile` 没有显式标记为 `semantic_guess`，会按默认 `safe_alias` 执行；这类通过组件名称自动补另一半轮廓的行为不应位于通用 AutoFix 中。当前 AutoFix 的审计也只记录修复前后整文档 Hash，而没有真实字段路径和旧新值。([GitHub][6])

应改为：

```text
fix_slot_half_profile
→ 移出通用AutoFix
→ 作为Knowledge-aware Repair Hint
→ 由LLM生成显式Patch
```

## 8.2 第二层：Validation Repair Loop

每轮 Repair Agent 必须同时看到：

```text
原始用户Prompt
当前RawGcadDocument
当前全部Validation错误
当前Knowledge Pack精简版
上轮Patch及拒绝原因
锁定的用户显式尺寸
允许修改的路径
```

Repair Agent可以修改：

* 局部点坐标；
* 圆弧方向；
* 半径；
* 操作参数；
* 输入输出引用；
* 局部操作顺序。

也可以在同一工程意图内更换建模方式，例如：

```text
radiusArc → threePointArc
统一fillet → 显式多个arc
完整轮廓 → 单侧+mirror
单次大布尔 → 分批布尔
```

这正是应当保留的 LLM 自由性。

但不得自动修改：

* 用户明确给定的尺寸；
* 齿数；
* 工作面定义；
* 连接类型；
* 安全等级；
* 用户要求的功能。

## 8.3 第三层：L2 Regeneration

5轮局部 Patch 仍失败，说明问题可能不是某个参数，而是整个 Feature Plan 不合理。

此时重新生成 RawGcadDocument，但必须带入失败记忆：

```text
前一方案使用了什么方法
失败节点
重复错误
已尝试的Patch
被拒绝的Patch
要求采用不同构造路线
```

不能只是重新发送原 Prompt，依赖随机性再生成一次。

## 8.4 第四层：Runtime Repair

Runtime Repair不能只允许改数值参数，也不能无限制改整个设计。

应分两类：

### Local Geometry Repair

允许：

* 缩小局部圆角；
* 修正弧方向；
* 延长刀具体；
* 调整布尔容差；
* 避免极短边；
* 替换等价圆弧构造方式。

### Structural Runtime Repair

允许：

* 分割一次复杂布尔；
* 改变局部操作顺序；
* 将一次阵列布尔改成批处理；
* 将后置圆角改成显式圆弧；
* 重建局部 Wire。

结构性 Repair 必须返回完整14阶段 Validation 和 Knowledge Compliance Validation。

若修复需要改变齿数、结构类型、主要尺寸或承载面语义，则不能自动执行，应当返回设计变更请求。

# 九、Governor 必须修正

当前 Governor 阶段表缺少：

* `root_terminal`
* `hole_semantics`

但14阶段 Validation 中确实包含这两个阶段。([GitHub][7])

不能手工维护两套顺序。应由 Validation Pipeline 导出：

```python
CORE_STAGE_ORDER = [
    *[name for name, _ in RAW_STAGES],
    "canonicalize",
    *[name for name, _ in CANONICAL_STAGES],
]
```

Governor 还必须记录：

```text
raw_graph_hash
canonical_graph_hash
error_signature_hash
patch_hash
rejected_patch_hash
rejection_reason
stage_rank
knowledge_pack_hash
```

当 `apply_patch()` 抛出 `ValueError` 时，可以跳过本轮，但必须：

1. 消耗本轮额度；
2. 保存被拒绝补丁的 Hash；
3. 把拒绝原因交给下一轮；
4. 防止再次生成相同补丁。

# 十、Knowledge Pack Loader 的确定性缺陷

当前 `_read_yaml()` 只接受 YAML `dict`：

```python
return data if isinstance(data, dict) else None
```

但 `known_conflicts.yaml` 是顶层列表。因此它在加载时先变为 `None`，后续再判断是否为 `list` 已经没有意义，最终 `known_conflicts` 永远为空。([GitHub][8])

应修改为：

```python
def _read_yaml_any(path: Path) -> Any:
    if not path.exists():
        return None
    with open(path, encoding="utf-8") as f:
        return yaml.safe_load(f)
```

然后：

```python
manifest = expect_dict(...)
rules = expect_dict(...)
known_conflicts = expect_list(...)
```

这非常重要，因为上传文档自身就同时出现了“两对大圆角半圆齿”和“三齿梯形榫头”两种描述，必须由冲突策略要求 LLM选择具体变体，而不能将二者混合。

# 十一、建议的代码修改位置

## `app/text-to-cad/server/main.py`

删除全局 `constraints_block` 中所有具体榫槽知识。

增加：

```python
knowledge_registry = KnowledgePackRegistry(...)
knowledge_summary = compile_l1_summary(...)
resolved = resolver.resolve(plan.selected_domain_skills)
knowledge_context = compile_l2_knowledge(resolved)
```

把 `knowledge_context` 注入 L2。

在 Validation 失败后真正接入：

```text
AutoFix
→ Repair Loop
→ L2 Retry
```

Runtime 后接入 Runtime Repair。

必须修改状态逻辑：

```python
if not rr.ok:
    status = "failed"
else:
    status = "completed"
```

当前代码即使 `rr.ok` 为 false，最后仍调用 `status="completed"`。([GitHub][1])

## `skills/orchestrator.py`

将 Knowledge Pack 摘要真正拼入 L1 Prompt，而不是只放在返回字典中。

`build_level2_authoring_prompt()` 增加：

```python
knowledge_context: str = ""
repair_context: str = ""
```

并将 L1 选择的 `selected_domain_skills` 传递到 L2。

## `knowledge/registry.py`

修复列表 YAML 加载。

增加：

* draft/reviewed/validated 运行策略；
* required dialect 检查；
* negative trigger 检查；
* applicable/not applicable 检查；
* Pack content hash。

目前 Registry 只明确拒绝 `deprecated`，Draft Pack仍可通过选择校验。([GitHub][8])

## `knowledge/resolver.py`

将 `token_budget` 改为真实 Token 预算，或者明确改名为 `char_budget`。当前实现用的是 `len(block)`，实际上按字符裁剪。([GitHub][9])

增加三种编译输出：

```text
author_context
validation_repair_context
runtime_repair_context
```

避免每次 Repair 都加载全部背景知识。

## `authoring/auto_fixer.py`

`semantic_guess` 和 `destructive` 默认禁用。

`fix_slot_half_profile`、默认位置、默认孔径等语义修改不能归类为 SafeFix。

# 十二、最终推荐架构

最终系统不应是：

```text
每种异形结构
→ 开发一个参数化模型
→ LLM填参数
```

也不应是：

```text
LLM自由猜坐标
→ OCC能运行就算成功
```

正确架构是：

```text
                  ┌──────────────────────┐
                  │ 专业Knowledge Pack   │
                  │ 语义/规则/策略/修复  │
                  └──────────┬───────────┘
                             ↓
用户需求 → LLM自由规划 → 通用GCAD操作图
                             ↓
                     通用几何验证
                             ↓
                AutoFix / LLM Repair
                             ↓
                       CadQuery/OCCT
                             ↓
                      Runtime Repair
                             ↓
                STEP + STL + 审计证据
```

其中 Knowledge Pack 只规定：

* 什么语义不能弄错；
* 哪些工程关系必须满足；
* 哪些构造方式值得优先；
* 哪些捷径禁止使用；
* 失败时应尝试哪些替代路线；
* 最终应检查什么。

它不规定：

* 必须使用固定数量的点；
* 必须使用固定特征树；
* 必须调用某个榫槽专用函数；
* 必须采用唯一参数化模板。

---

# 十三、验收标准

修复完成后至少应满足：

1. 普通支架请求绝不会收到榫槽 Knowledge。
2. 榫槽请求必须记录所使用的 Pack ID、版本和 Hash。
3. Draft Pack 是否允许执行由运行环境明确控制。
4. Pack冲突必须在 L2 前解决。
5. 14阶段失败后确实进入5轮 Repair。
6. 每个被接受和拒绝的 Patch 都有字段级记录。
7. Repair 每轮都能看到专业规则。
8. L2 Retry明确采用不同构造方案。
9. Runtime Repair 的每个补丁都真正重新运行一次 OCC。
10. Runtime失败绝不标记为完成。
11. 槽型可以由不同点数、不同操作序列生成。
12. 系统能够生成标准三齿、两齿大圆角和用户自定义混合齿形，而无需新增三个专用参数化模型。
13. 所有结果通过闭合、自交、半径、相切、对称、阵列数量和 B-Rep 合法性检查。
14. Knowledge Pack 中的 Prompt 规则可被版本化、审计和复现。

## 最终结论

你的核心路线不需要推翻：

> **LLM自由建模 + 专业Prompt约束 + 14阶段验证 + AutoFix + LLM Repair + Runtime Repair，才是适合大量异形工程结构的可扩展方案。**

需要修复的不是“缺少一个榫槽参数化求解器”，而是：

1. Knowledge Pack 尚未真正接入主链路；
2. 专业Prompt存在若干错误和过度刚性规则；
3. 三级Repair模块没有在当前Web入口形成闭环；
4. AutoFix、Governor和Runtime状态治理还不够严格；
5. 缺少能够为所有异形结构复用的通用几何反馈。

因此最终原则应定为：

> **不为每一个异形部位编写专用参数化模型；让LLM负责生成拓扑和建模路线，让Knowledge Pack负责专业约束，让通用几何验证提供客观错误，让Repair Loop负责迭代收敛。**
