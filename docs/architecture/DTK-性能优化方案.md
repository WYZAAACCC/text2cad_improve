# DTK 架构优化方案：在 DeepSeek 上保持领先

**依据**：8 Agent × 2 场景的精确诊断数据（Token、缓存率、每步延迟、质量评分）
**目标**：在不牺牲质量的前提下，Token 效率提升 40%，延迟降低 25%

---

## 一、诊断数据揭示的四个根因

### 根因 1：API 调用次数过多（3-6 次 vs LangChain 的 2 次）

| 框架 | 金融分析 API 调用 | Prompt Token | 工具调用/步 |
|------|:---:|------:|:---:|
| LangChain | **2 次** | 4,764 | 全部在第 1 步 |
| DTK Fast | 3 次 | 5,955 | 3+16+0 |
| DTK Stable | 6 次 | 20,239 | 3+13+3+3+1+0 |

**每多一次 API 调用，成本增加约 30%**：额外的 prompt token（累积历史）+ 网络延迟（2-3s）+ thinking token（Stable 模式）。DTK Stable 的 6 次调用中，Step#3-5（工具调用 3+3+1）是在"挤牙膏"——每次只调 1-3 个工具，白白消耗了 3 轮 API 请求。

### 根因 2：Thinking 模式的 Token 膨胀

```
DTK Stable Step#1: P:1199 C:408  → 推理 token 隐藏在 prompt 中
DTK Fast   Step#1: P:1149 C:217  → 无推理 token
```

Stable 模式的 `completion_tokens` 看起来更多（408 vs 217），但实际上 DeepSeek 的推理 token 不计入 `completion_tokens`，而是计入 `prompt_tokens`（作为下一轮的 reasoning_content 传回）。**6 次调用 × 每次约 400 reasoning token = 2,400 额外 prompt token**。这是 Stable 模式总 prompt 高达 20,239 的原因。

### 根因 3：缓存命中率的"最后一公里"

| 框架 | 金融缓存率 | 为什么 |
|------|:---:|------|
| LangChain | **99%** | 2 次调用，第 1 次缓存 100%，第 2 次 99% |
| DTK Fast | 77% | 第 1 次 89%，第 2 次 80%，第 3 次 72% |
| DTK Stable | 91% | 首次 96%，后续逐步下降到 67% |

**缓存命中率随步骤下降**：因为 DTK 的每步都在系统提示后追加了新的动态内容（工具结果），导致 prefix 逐次变长。CacheStabilizer 冻结了系统提示本体，但**工具结果的追加位置**在系统提示和早期对话之间，破坏了字节对齐。

### 根因 4：工具调用策略的"增量式困境"

```
DTK Fast 金融分析:
  Step#1: 调 3 个工具 → 发现不够
  Step#2: 调 16 个工具 → 几乎全部
  Step#3: 最终合成

DTK Fast 供应链:
  Step#1: 调 6 个工具 → 不够
  Step#2: 调 3 个工具 → 还不够
  Step#3: 调 2 个 → ...
  Step#6: 还在调工具 → max_steps 截断！
```

模型在 Fast 模式下不知道自己需要多少数据，于是在**探索-利用困境**中选择了探索（继续调工具）而非利用（开始合成）。这是所有 agent 框架的经典问题，但 DTK 对此特别脆弱——因为没有给模型任何"何时停止"的信号。

## 二、五个架构改进方案

### 改进 1：引入"两步协议"——消除多余的 API 调用 ⭐⭐⭐

**当前行为**：模型每轮自主决定调多少工具，可能 1-6 个不等，导致不可预测的 API 调用次数。

**改进方案**：在 Agent 循环中插入一个隐式的"计划阶段"。模型在第一次回复中只输出**工具调用计划**（不调工具），框架解析计划后，第二轮一次性执行所有工具并合成最终答案。

```
改进前：Step1(3工具) → Step2(16工具) → Step3(最终)  = 3 次 API
改进后：Step1(计划19工具) → Step2(执行+合成)         = 2 次 API
```

**实现位置**：`runtime.py` 的 `chat()` 循环第一轮特殊处理——第一轮 system prompt 追加：

```
在开始分析之前，先列出你需要的所有工具调用计划（只需列出工具名和参数概
要，不需要实际参数），之后一次性执行所有工具并给出最终分析。
```

**预期效果**：API 调用数 -40%，总 prompt token -35%。

### 改进 2：推理 Token 回收机制 ⭐⭐⭐

**当前行为**：Thinking 模式的推理内容自动传回下一轮，导致 prompt token 膨胀。

**改进方案**：对 reasoning_content 做"压缩回传"——不是传回完整推理文本，而是解析出关键决策点（类似 Reasonix 的 Thought Harvesting），仅回传结构化摘要。

```python
# 改进前：完整 reasoning_content 传回
assistant_msg["reasoning_content"] = response.reasoning_content  # 400-800 tokens

# 改进后：压缩为结构化摘要
assistant_msg["reasoning_content"] = compress_reasoning(response.reasoning_content)
# → "Plan: call web_search×3, calculate_roi×3, risk_score×3. Findings: A high risk, B stable."
# → 50-100 tokens
```

关键洞察：DeepSeek 只要求传回 reasoning_content 的**存在**以保持对话格式一致性，不要求**完整的字节相同**。压缩后的推理摘要既满足 API 要求，又减少 70-80% 的推理 token 传输。

**实现位置**：`runtime.py` 中构建 assistant_msg 时插入 `_compress_reasoning()`。

**预期效果**：Stable 模式 prompt token -40%，总成本 -35%。

### 改进 3：缓存对齐——工具结果追加到尾部而非前缀后 ⭐⭐

**当前行为**：工具结果追加到 assistant 消息后面，位于系统提示和早期对话之间。

```
[system] [user] [assistant+tool_calls] [tool_result_1] [tool_result_2] ...
                                            ↑ 插入在中间，破坏缓存对齐
```

**改进方案**：将工具结果追加到对话**末尾**而非前缀后。DeepSeek 缓存的是从第一条消息开始的连续匹配前缀。如果工具结果插入在中间，前缀被截断。

```
改进后：
[system] [user] [assistant+tool_calls] [assistant_final] [tool_results_appendix]
                                              ↑ 完整的前缀缓存下来   ↑ 新内容在末尾
```

实现：在构建 assistant 消息时，不在中间插入 tool results，而是把 tool results 收集到一个附录块，放在最终 assistant 回复之后。

**实现位置**：`runtime.py` 的 `chat()` 循环中修改 working_messages 的构建顺序。

**预期效果**：缓存率从 72-77% 提升到 90-95%。

### 改进 4：自适应 max_steps —— 按场景复杂度自动调整 ⭐⭐

**当前行为**：`max_steps=6` 对所有场景一视同仁。简单场景浪费配额，复杂场景（供应链）被截断。

**改进方案**：根据用户任务长度和工具数量，动态估算最小步数。公式：

```python
estimated_min_steps = max(3, len(task_chars) // 2000 + len(tools) // 4)
self._max_steps = min(estimated_min_steps * 2, 10)  # 2x 安全边际，上限 10
```

更精确的方案：在第一次 API 调用后，根据模型的工具调用数量调整。如果第一次调用了超过 10 个工具，说明是复杂场景，自动提升 max_steps。

**实现位置**：`runtime.py` 的 `__init__` 或 `chat()` 的第一次迭代后。

**预期效果**：供应链不再截断，简单场景减少配额浪费。

### 改进 5：早停信号——教会模型"何时停止调工具" ⭐

**当前行为**：模型在 Fast 模式下不知道何时该停止调用工具、开始合成答案。

**改进方案**：在 system prompt 中明确加入"早停规则"：

```
工具调用策略：
1. 第一轮：列出所有你能想到的需要调用的工具，一次性全部调用
2. 第二轮：如果你发现了新的必要数据需求，再补充调用一次
3. 第三轮：必须开始合成最终答案，即使数据不完全
记住：完美是好的敌人。三轮之后强制停止。
```

同时在 `runtime.py` 中，当 `step >= max_steps - 2` 时，自动追加一个提示：

```
你只剩最后一轮回复机会了。请在下一轮中直接给出最终答案，
不要再调用新工具。基于已有数据进行最佳判断。
```

**实现位置**：system prompt 模板 + `runtime.py` 倒数第二步。

**预期效果**：供应链场景的 DTK 不再截断，完成率 100%。

## 三、预期效果汇总

基于诊断数据的线性推估：

| 指标 | DTK Fast (当前) | DTK Fast (优化后) | 改善 |
|------|:---:|:---:|:---:|
| API 调用次数 | 3-6 | 2-3 | **-40%** |
| Prompt Token | 5,955 | ~4,000 | **-33%** |
| 总 Token | 9,798 | ~6,500 | **-34%** |
| 缓存率 | 77% | ~92% | **+19%** |
| 延迟 | 56s | ~35s | **-38%** |
| 成本 | ¥0.00133 | ~¥0.00080 | **-40%** |
| 质量 | 7.5 | 7.5+ | 持平或提升 |

| 指标 | DTK Stable (当前) | DTK Stable (优化后) | 改善 |
|------|:---:|:---:|:---:|
| API 调用次数 | 6 | 2-3 | **-60%** |
| Prompt Token | 20,239 | ~8,000 | **-60%** |
| 总 Token | 25,669 | ~12,000 | **-53%** |
| 延迟 | 92s | ~50s | **-46%** |
| 成本 | ¥0.00203 | ~¥0.00100 | **-51%** |

**优化后 DTK 对比 LangChain**：
- Fast 模式：成本从 29% 劣势 → **22% 优势**（¥0.00080 vs ¥0.00103）
- Stable 模式：成本从 97% 劣势 → **持平**（¥0.00100 vs ¥0.00103）
- 质量：Stable 模式已有多次达到 8.3-8.5，保持领先

| 优先级 | 改进 | 工作量 | 预期收益 |

| **P0** | 改进 4：自适应 max_steps | 30 分钟 | 消除截断问题 |
| **P1** | 改进 1：两步协议 | 2 小时 | Token -35%，延迟 -40% |
| **P1** | 改进 3：缓存对齐 | 1 小时 | 缓存率 +15-20% |

**建议执行顺序**：P0 先行（消除截断）→ P1 跟进（核心效率提升）→ P2 收尾（Stable 模式优化）
