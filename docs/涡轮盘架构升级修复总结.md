## 三、深度审查发现的遗留问题

第一轮修复完成后，作为严苛的代码审查员重新审视所有修复，发现 **3 个遗留问题**，导致 fail-closed 策略在某些场景下被绕过：

### 问题 A：Z 重叠检测在错误的文件中（严重）

**位置**：`bases/axisymmetric/preflight.py:81-102`

**根因**：v0.2 dialect 系统的 validation pipeline **只调用** `dialects/axisymmetric/dialect.py` 的 `preflight_component`（通过 `validation/geometry_preflight.py:74`）。而 `bases/axisymmetric/preflight.py` 的 `AXISYMMETRIC_PREFLIGHT_HANDLERS` 只在 `bases/axisymmetric/runner.py`（v0.1 legacy 系统）中使用，v0.2 流程不会调用 runner.py。

**影响**：第一轮添加到 `preflight.py` 的 Z 重叠检测**永远不会被执行**，是死代码。

### 问题 B：handlers.py 的"区域描述"检测条件过严（中等）

**位置**：`dialects/axisymmetric/handlers.py:60-69`

**根因**：检测条件是"所有 z_front 相同 **AND** 所有 z_rear 相同"：
```python
if len(z_fronts) == 1 and len(z_rears) == 1:
    raise ValueError(...)
```

但实际 LLM 输出的区域描述中，每个区域 Z 区间不同（bore_bottom: z=5→6, bore_wall: z=6→115, hub_front_face: z=5→42.5, ...），`z_fronts` 有 4 个不同值，**不会触发检测**。

### 问题 C：_fix_profile_stations 检测条件过严（中等）

**位置**：`authoring/auto_fixer.py:711`

**根因**：与问题 B 相同的条件 `len(z_fronts) == 1 and len(z_rears) == 1`，不会触发实际场景。

### 核心风险

如果 LLM 修复了 A005 共面问题（移除 r=60 站点）但保留区域描述风格（不同 Z 区间）：
1. A005 不再触发（无 r ≈ bore_radius 的站点）
2. Z 重叠检测不触发（死代码）
3. handlers.py 区域描述检测不触发（条件过严）
4. sort-by-Z 算法 silently 产生自相交多边形
5. CadQuery revolve 可能失败，或产生错误几何体
6. **fail-closed 策略被绕过**

## 四、第二轮修复（4 个修复项）

### 修复 1：将 Z 重叠检测移到 dialect.py 的 preflight_component（问题 A）

**文件**：[dialect.py:274-300](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/dialect.py#L274-L300)

**位置**：`preflight_component` 方法，A005 检测之后、`return ValidationReport` 之前

**添加内容**：
```python
# ── A006: revolve_profile Z 重叠检测 ──
# 检测"区域描述"风格：多个 station 的 Z 区间重叠
# 这是 LLM 把 bore/hub/web/rim 分区域输出的典型模式，违反 Z 单值约束
for n in nodes:
    if n.op != "revolve_profile":
        continue
    ps = n.typed_params.get("profile_stations") or n.params.get("profile_stations", [])
    if len(ps) < 2:
        continue
    z_ranges = [(s.get("z_front_mm", 0), s.get("z_rear_mm", 0)) for s in ps]
    overlap_count = 0
    for i, (zf1, zr1) in enumerate(z_ranges):
        for j, (zf2, zr2) in enumerate(z_ranges[i+1:], i+1):
            if zf1 < zr2 and zf2 < zr1:  # Z 区间严格重叠
                overlap_count += 1
    if overlap_count > 0:
        issues.append(ValidationIssue(
            stage=stage, code="a006_profile_z_overlap",
            message=(
                f"revolve_profile stations have {overlap_count} overlapping Z ranges. "
                f"This indicates regional description style (bore/hub/web/rim), "
                f"which violates the Z single-valued outer contour constraint. "
                f"Convert to outer contour continuous segments where each Z position "
                f"has exactly one radius."
            ),
            severity="error", node_id=n.id,
        ))
```

**why**：将 Z 重叠检测放到正确的 validation pipeline 调用路径上，使其在 v0.2 系统中实际执行。

## 五、四层防线架构

修复完成后，系统建立了完整的四层防线，确保 fail-closed 策略不被绕过：

```
┌─────────────────────────────────────────────────────────────────┐
│ Layer 1: Prompt 层（引导）                                       │
│  ├── system prompt（方言约束 + 安全声明 + 操作契约）             │
│  ├── schema description（字段语义说明）                          │
│  ├── few-shot examples（TURBINE_DISC_REFERENCE_EXAMPLE）         │
│  └── anti-example（regional_profile_stations_bad）               │
├─────────────────────────────────────────────────────────────────┤
│ Layer 2: Autofix 层（自动修复 + 标记）                           │
│  ├── _fix_slot_station_order（slot 深度排序）                    │
│  ├── _fix_root_node（优先 required=True 节点）                   │
│  ├── _fix_profile_stations（检测 Z 重叠 → 标记 _autofix_hints）  │
│  └── _fix_chamfer_fillet_optional（标记 optional）               │
├─────────────────────────────────────────────────────────────────┤
│ Layer 3: Preflight 层（validation pipeline 拦截）                │
│  ├── A001: stations_count, radius, z_order                       │
│  ├── A002: center_bore diameter                                  │
│  ├── A003: circular_hole_pattern                                 │
│  ├── A004: annular_groove                                        │
│  ├── A005: bore-vs-profile coplanar detection                    │
│  └── A006: revolve_profile Z overlap detection ← 新增            │
├─────────────────────────────────────────────────────────────────┤
│ Layer 4: Handler 层（runtime fail-closed）                       │
│  ├── handle_revolve_profile（检测 Z 重叠 → raise ValueError）    │
│  ├── handle_cut_annular_groove（side 语义修正）                  │
│  └── handle_apply_safe_chamfer（降级策略）                       │
└─────────────────────────────────────────────────────────────────┘
```

### 防线调用路径

```
LLM 输出
  ↓
Route → Context → FeatureSequence → NodeParams → Validate
                                                    ↓
                                              AutoFix（Layer 2）
                                                    ↓
                                              Preflight（Layer 3）
                                                    ↓ fail-closed
                                              Runtime（Layer 4）
                                                    ↓ fail-closed
                                              生成 STEP 文件
```

## 八、设计决策记录

### 决策 1：修复范围

**选择**：架构核心全修复（Prompt + Schema + Handler + 配置）

**理由**：不是只让涡轮盘成功生成，而是找出某一类问题为什么会出现，区分架构原因 vs LLM 原因，对架构和代码进行升级让整体性能提升。

### 决策 2：revolve_profile 策略

**选择**：保持 IR 单值语义 + 检测报错 + 引导重试

**理由**：不修改 IR 语义，保持 sort-by-Z 算法的简洁性。通过多层检测确保错误的"区域描述"风格被拦截，引导 LLM 重新生成正确的外轮廓连续段。

### 决策 3：handler 检测 vs preflight 检测的重复

**选择**：保留两者

**理由**：preflight 是第一道防线（validation 阶段），handler 是最后防线（runtime 阶段）。即使 preflight 漏检，handler 也会 fail-closed。

### 决策 4：autofix 策略

**选择**：保持"标记 _autofix_hints 并跳过"，不直接修改

**理由**：区域描述风格无法安全自动转换为连续段，需要 LLM repair 处理。

### 决策 5：删除 vs 保留 bases/axisymmetric/preflight.py 的死代码

**选择**：删除

**理由**：这是第一轮添加的错误代码，属于"自己的 mess"。按照 karpathy-guidelines 的"clean up only your own mess"原则清理。
