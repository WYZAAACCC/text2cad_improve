# 涡轮盘修复方案 — 通用性与影响范围规划

**日期**: 2026-07-07
**目标**: 修复涡轮盘生成失败，但**所有修复必须是通用的**，不能为涡轮盘打特殊补丁，不能影响其他零件（齿轮、轴承、壳体、法兰、支架等）的构建。
**核心原则**: 每个修复都应使系统对所有零件都更健壮，而不是仅对涡轮盘生效。

---

## 一、修复通用性分析框架

### 1.1 评判标准

每个修复方案必须满足以下三条通用性标准：

| 标准 | 说明 |
|------|------|
| **A. 规则一致性** | 修复后的行为必须与已有验证规则一致，不引入新规则冲突 |
| **B. 不误伤合法输出** | 不能改变合法 LLM 输出的处理结果 |
| **C. 不针对特定零件** | 不能包含任何零件类型判断（如 `if "turbine" in prompt`） |

### 1.2 现有合法 root_node 模式（基线）

通过对 [base_packages/](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/base_packages/) 和测试用例的统计，**所有合法零件的 root_node 都指向 solid-producing 节点**：

| 零件类型 | root_node 指向 | op | output_types |
|----------|----------------|-----|--------------|
| 垫圈 (washer) | `n_chamfer` | apply_safe_chamfer | `["solid"]` |
| 圆盘 (disc) | `n_slots` | cut_rim_slot_pattern | `["solid"]` |
| 草图旋转盘 | `n_revolve` | revolve_profile | `["solid"]` |
| 装配体 | `n_final` | boolean_cut/union | `["solid"]` |
| 板件 (plate) | `n_fillet` | apply_safe_fillet | `["solid"]` |
| 装配体 | `n_union` | boolean_union | `["solid"]` |
| 轴对称件 | `n_bore` | cut_center_bore | `["solid"]` |
| 草图拉伸件 | `n_se` | extrude_rectangle | `["solid"]` |

**关键事实**: 没有任何合法零件的 root_node 指向 profile-producing 节点（如 close_profile, add_polyline, add_arc_segment）。

### 1.3 op 的 output_types 分类

| 类型 | op 列表 | 能否作为 root_node |
|------|---------|-------------------|
| **Solid-producing** | extrude_rectangle, extrude_profile, revolve_profile (sketch_profile), revolve_profile (axisymmetric), cut_*, add_*, apply_safe_*, boolean_*, translate_solid, rotate_solid, place_component, circular_pattern_component, linear_pattern_component, cut_profile, fillet_sketch(?), mirror_profile(?) | ✅ 可以 |
| **Profile-producing** | create_2d_sketch(→sketch), add_line_segment, add_arc_segment, add_circle, add_polyline, add_slot, close_profile | ❌ 不可以 |
| **Sketch-producing** | create_2d_sketch | ❌ 不可以 |

## 二、修复方案通用性逐项分析

### 2.1 P0-1: 修复 `_fix_root_node` 验证输出类型

**修复位置**: [auto_fixer.py:573-611](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L573-L611)

**当前逻辑**:
```python
if root_node_obj and root_node_obj.get("required", True):
    continue  # required=True 就保留，不检查输出类型
```

**修复后逻辑**:
```python
if root_node_obj and root_node_obj.get("required", True):
    has_body_solid = any(
        o.get("name") == "body" and o.get("type") == "solid"
        for o in root_node_obj.get("outputs", [])
    )
    if has_body_solid:
        continue  # required=True 且输出 body:solid，保留
    # fall through 重新选择
```

#### 通用性评估

| 标准 | 评估 | 说明 |
|------|------|------|
| **A. 规则一致性** | ✅ 通过 | 与 [structure.py:57-60](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/structure.py#L57-L60) 的 root_node 验证规则一致 |
| **B. 不误伤合法输出** | ✅ 通过 | 见下方详细分析 |
| **C. 不针对特定零件** | ✅ 通过 | 不包含任何零件类型判断 |

#### 不误伤合法输出分析

**Case 1: 合法的 root_node 指向 solid-producing 节点**
- 修复前: `required=True` + `outputs=[body:solid]` → `continue` 保留
- 修复后: `required=True` + `has_body_solid=True` → `continue` 保留
- **行为不变** ✅

**Case 2: LLM 错误地将 root_node 指向 profile-producing 节点（如 close_profile）**
- 修复前: `required=True` → `continue` 保留 → structure 验证失败
- 修复后: `required=True` + `has_body_solid=False` → fall through 重新选择 → 找到 body:solid 节点 → structure 验证通过
- **行为改善** ✅

**Case 3: root_node 指向的节点 required=False**
- 修复前: 已有逻辑会重新选择
- 修复后: 同样重新选择
- **行为不变** ✅

**Case 4: root_node 指向的节点不存在于 nodes 中**
- 修复前: 已有逻辑会重新选择
- 修复后: 同样重新选择
- **行为不变** ✅

**Case 5: 组件中没有任何 body:solid 节点**
- 修复前: `required=True` 保留无效 root_node → structure 失败
- 修复后: fall through 重新选择 → `body_nodes` 为空 → `comp["root_node"] = comp_nodes[-1]["id"]` → structure 仍失败
- **行为不变**（仍然失败，但这是正确的 — 没有-solid 的组件本就应该失败）✅

#### 潜在风险与缓解

**风险 1: `fix_output_names` 修改 outputs 后，`_fix_root_node` 看到的是修改后的 outputs**

- `fix_output_names` (step 9) 在 `_fix_root_node` (step 20) 之前运行
- 如果 LLM 伪造了 `outputs=[body:solid]`，`fix_output_names` 会按 spec 修正为 `[profile:profile]`
- 然后 `_fix_root_node` 看到修正后的 outputs，正确识别为无效 root_node
- **这是预期行为**，不是风险 ✅

**风险 2: 重新选择 root_node 时，可能选到非终端 solid 节点**

- `_fix_root_node` 的重新选择逻辑: `comp["root_node"] = required_body_nodes[-1]["id"]`
- 这选的是组件中最后一个 required=True 且输出 body:solid 的节点
- 但这可能是非终端节点（被下游节点消费）
- **缓解**: 这是 `root_terminal` 验证阶段（stage 2）的责任，不是 `_fix_root_node` 的责任
- 如果选到非终端节点，`root_terminal` 会报 `ROOT_NOT_TERMINAL_SOLID` 错误
- **这是设计分工**，不是 bug ✅

#### 结论

**P0-1 是完全通用的修复**，符合所有三条通用性标准。它只是让 `_fix_root_node` 更严格地遵循已有的 structure 验证规则，不引入任何新规则，不影响任何合法输出。

### 2.2 P0-3: L2 prompt 添加禁止 passthrough 节点的规则

**修复位置**: [prompts.py:72-171](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/skills/prompts.py#L72-L171)

**新增规则**:
```
34. Do NOT create passthrough/marker nodes at the end of a component.
    The root_node must directly point to a solid-producing node
    (extrude_profile, revolve_profile, boolean_cut, boolean_union, etc.).
35. close_profile may ONLY follow add_polyline/add_line_segment/add_arc_segment/add_circle.
    close_profile MUST NOT follow extrude_profile, revolve_profile, or any solid-producing op.
```

#### 通用性评估

| 标准 | 评估 | 说明 |
|------|------|------|
| **A. 规则一致性** | ✅ 通过 | 与 [structure.py:57-60](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/structure.py#L57-L60) 和 [typecheck.py:38-44](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/typecheck.py#L38-L44) 的规则一致 |
| **B. 不误伤合法输出** | ✅ 通过 | 见下方详细分析 |
| **C. 不针对特定零件** | ✅ 通过 | 规则适用于所有零件 |

#### 不误伤合法输出分析

**Case 1: 合法的草图 → 闭合 → 拉伸/旋转 流程**
- `create_2d_sketch → add_polyline → close_profile → extrude_profile/revolve_profile`
- close_profile 的输入是 profile（来自 add_polyline），符合规则 35
- root_node 指向 extrude_profile/revolve_profile（solid-producing），符合规则 34
- **不受影响** ✅

**Case 2: 合法的轴对称流程**
- `revolve_profile → cut_center_bore → cut_annular_groove → apply_safe_chamfer`
- 没有 close_profile，符合规则 35
- root_node 指向 apply_safe_chamfer（solid-producing），符合规则 34
- **不受影响** ✅

**Case 3: 合法的草图拉伸流程**
- `extrude_rectangle → cut_hole → apply_safe_fillet`
- 没有 close_profile，符合规则 35
- root_node 指向 apply_safe_fillet（solid-producing），符合规则 34
- **不受影响** ✅

**Case 4: 合法的装配流程**
- `place_component(disc) → place_component(cutter) → circular_pattern_component → boolean_cut`
- 没有 close_profile，符合规则 35
- root_node 指向 boolean_cut（solid-producing），符合规则 34
- **不受影响** ✅

#### 潜在风险与缓解

**风险 1: 规则 35 是否过于严格？**

- 规则 35 禁止 close_profile 跟在 solid-producing op 后面
- 但 close_profile 的 spec 是 `input_types=["profile"]`，本来就不能接收 solid 输入
- 所以规则 35 只是显式说明了 spec 已经隐含的约束
- **不是新规则，只是显式化** ✅

**风险 2: 是否有零件需要在 solid 后面修改 profile？**

- 从 op 的 input_types 看，没有任何 profile-producing op 能接收 solid 输入
- 所以 "solid → profile" 的流程本来就是非法的
- 规则 34 只是显式说明了这一点
- **不是新规则，只是显式化** ✅

#### 结论

**P0-3 是完全通用的修复**，只是将已有的隐含约束（op spec 的 input_types/output_types）显式化为 prompt 规则，帮助 LLM 更好地理解。不影响任何合法输出。

### 2.3 P0-2: 增加 passthrough 节点检测与清理

**修复位置**: `auto_fixer.py` 新增 `_strip_passthrough_nodes` 函数

**逻辑**: 检测节点的实际输入类型与 op spec 的 input_types 不匹配，删除 passthrough 节点并将下游引用重定向到其输入源。

#### 通用性评估

| 标准 | 评估 | 说明 |
|------|------|------|
| **A. 规则一致性** | ✅ 通过 | 与 [typecheck.py:38-44](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/typecheck.py#L38-L44) 的 input_type_mismatch 规则一致 |
| **B. 不误伤合法输出** | ⚠️ 需要谨慎实现 | 见下方详细分析 |
| **C. 不针对特定零件** | ✅ 通过 | 不包含任何零件类型判断 |

#### 不误伤合法输出分析（关键风险点）

**Case 1: 合法的 profile → profile 链**
- `add_polyline → close_profile`
- close_profile spec: `input_types=["profile"]`
- add_polyline outputs: `[profile:profile]`
- 输入类型匹配 → **不删除** ✅

**Case 2: 合法的 solid → solid 链**
- `extrude_rectangle → cut_hole → apply_safe_fillet`
- cut_hole spec: `input_types=["solid"]`
- extrude_rectangle outputs: `[body:solid]`
- 输入类型匹配 → **不删除** ✅

**Case 3: 合法的 solid + profile → solid 链（cut_profile）**
- `extrude_rectangle → create_2d_sketch → add_polyline → cut_profile`
- cut_profile spec: `input_types=["solid", "profile"]`
- 第一个输入是 solid（来自 extrude_rectangle），第二个是 profile（来自 add_polyline）
- 输入类型匹配 → **不删除** ✅

**Case 4: 非法的 solid → profile 链（passthrough）**
- `revolve_profile → close_profile` (LLM 误用)
- close_profile spec: `input_types=["profile"]`
- revolve_profile outputs: `[body:solid]`
- 输入类型不匹配 → **删除 close_profile 节点，下游引用重定向到 revolve_profile** ✅

#### 实现关键点（避免误伤）

1. **必须基于 spec 的 input_types 判断**，不能基于 op 名称黑名单
   - ❌ 错误: `if op == "close_profile" and input_is_solid: delete`
   - ✅ 正确: `if input_type not in spec.input_types: delete`

2. **必须正确解析输入类型**
   - 需要查找上游节点的 outputs 来确定实际输入类型
   - 不能假设输入类型

3. **删除后必须修复下游引用**
   - 将下游节点的 `inputs[i].node` 从 passthrough 节点改为 passthrough 的输入源
   - 将 `inputs[i].output` 从 passthrough 的输出名改为输入源的输出名

4. **必须处理多个 passthrough 节点的链式情况**
   - `A(solid) → B(passthrough) → C(passthrough) → D(solid)`
   - 删除 B 和 C 后，D 的输入应指向 A

#### 潜在风险与缓解

**风险 1: 误删合法的类型转换节点**

- 当前 IR 中没有合法的 "solid → profile" 转换 op
- 所有 op 的 input_types 都明确规定了输入类型
- 所以基于 spec input_types 的判断不会误删合法节点
- **风险已缓解** ✅

**风险 2: 下游引用重定向错误**

- 如果 passthrough 节点的输入源有多个 outputs，重定向时可能选错
- **缓解**: 只重定向到 passthrough 节点的第一个输入（passthrough 节点通常只有一个输入）

**风险 3: 删除节点后，root_node 指向被删除的节点**

- **缓解**: `_fix_root_node` 在 `_strip_passthrough_nodes` 之后运行，会重新选择 root_node

#### 实施建议

**建议: P0-2 暂缓实施，先实施 P0-1 + P0-3**

- P0-1 已经能解决本次测试失败（重新选择 root_node）
- P0-3 能从源头防止 LLM 创建 passthrough 节点
- P0-2 是兜底清理，实现复杂度高，风险较大
- 如果 P0-1 + P0-3 足够，可以不实施 P0-2

#### 结论

**P0-2 是通用修复，但实现复杂度高，建议暂缓**。如果实施，必须严格基于 spec input_types 判断，不能基于 op 名称黑名单。

### 2.4 P1-1: circular_pattern 使用 count 阈值切换策略

**修复位置**: [composition/handlers.py:179-222](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/composition/handlers.py#L179-L222)

**当前逻辑**: 60 个 cutter 的 union 导致 OCCT 拓扑损坏

**修复方案**: count 阈值切换策略

#### 通用性评估

| 标准 | 评估 | 说明 |
|------|------|------|
| **A. 规则一致性** | ✅ 通过 | 与 [axisymmetric/handlers.py:259-299](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/handlers.py#L259-L299) 的 cut_rim_slot_pattern count > 6 策略一致 |
| **B. 不误伤合法输出** | ✅ 通过 | 见下方详细分析 |
| **C. 不针对特定零件** | ✅ 通过 | 阈值基于 count，不基于零件类型 |

#### 不误伤合法输出分析

**Case 1: 少量 cutter（count ≤ 6，如 4 孔法兰、6 孔螺栓盘）**
- 修复前: union 策略
- 修复后: union 策略（count ≤ 6 不变）
- **行为不变** ✅

**Case 2: 大量 cutter（count > 6，如涡轮盘 60 槽）**
- 修复前: union 策略 → 60 次 union → 拓扑损坏
- 修复后: sequential 策略 → 60 次 cut → 拓扑稳定
- **行为改善** ✅

#### 实现关键点

**问题**: `circular_pattern_component` 的输出是 patterned cutters（solid），作为 `boolean_cut` 的 tool 输入。如果要改用 sequential cut，需要：

**方案 A**: 在 `circular_pattern_component` 中根据 count 切换策略
- count ≤ 6: union 所有 cutters 成一个 compound，返回 solid
- count > 6: 返回 cutters 列表（但 IR 中没有 "solids list" 类型）

**方案 B**: 在 `boolean_cut` 中检测 tool 是否是 circular_pattern 的输出
- 如果是且 count > 6，改为 sequential cut
- 但这需要 boolean_cut 知道 tool 的来源 op，违反了 IR 的抽象

**方案 C**: 新增 `cut_circular_pattern` op
- 专门用于 "circular pattern + cut" 的组合操作
- 内部根据 count 切换策略
- 但这增加了 IR 的复杂度

**推荐方案**: 方案 A + 在 `circular_pattern_component` 中返回 compound

- count ≤ 6: union 所有 cutters 成一个 solid，返回 body:solid
- count > 6: 不 union，直接 compound 所有 cutters，返回 body:solid（compound 也是 solid）
- `boolean_cut` 的 handler 已能处理 compound tool（CadQuery 的 `target.cut(compound)` 会逐个 cut）

**验证**: 需要确认 CadQuery 的 `target.cut(compound)` 是否会逐个 cut。如果不支持，需要在 `boolean_cut` handler 中遍历 compound 的 solids 逐个 cut。

#### 潜在风险与缓解

**风险 1: compound 作为 tool 传入 boolean_cut**

- 如果 `boolean_cut` handler 不支持 compound tool，会失败
- **缓解**: 在 `boolean_cut` handler 中检测 tool 是否是 compound，如果是则遍历 solids 逐个 cut

**风险 2: count 阈值 6 是否合适**

- 与 [axisymmetric/handlers.py:259-299](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/handlers.py#L259-L299) 的 cut_rim_slot_pattern 一致
- 是已有约定，不是新阈值
- **风险已缓解** ✅

#### 结论

**P1-1 是通用修复**，与已有的 cut_rim_slot_pattern 策略一致。但实现需要谨慎，建议先验证 CadQuery 对 compound tool 的支持。

### 2.5 P1-2: 引入 LLM repair loop 在 L2 阶段

**修复位置**: [main.py:430-447](file:///e:/auto_detection_process/app/text-to-cad/server/main.py#L430-L447)

**修复方案**: 在 autofix 失败后，调用 LLM repair loop（使用已有的 `RepairPatchV2` 和 `governor.can_repair_v2`，max 3 attempts）

#### 通用性评估

| 标准 | 评估 | 说明 |
|------|------|------|
| **A. 规则一致性** | ✅ 通过 | 使用已有的 repair 框架 |
| **B. 不误伤合法输出** | ✅ 通过 | 只在 validation 失败时触发 |
| **C. 不针对特定零件** | ✅ 通过 | 适用于所有零件 |

#### 不误伤合法输出分析

**Case 1: 验证通过的合法输出**
- 修复前: 直接进入 runtime
- 修复后: 直接进入 runtime（repair loop 只在验证失败时触发）
- **行为不变** ✅

**Case 2: 验证失败但 autofix 修复成功**
- 修复前: autofix 后重新验证通过 → 进入 runtime
- 修复后: 同样（repair loop 只在 autofix 后仍失败时触发）
- **行为不变** ✅

**Case 3: autofix 后仍失败**
- 修复前: 直接 fail
- 修复后: 调用 LLM repair loop（max 3 attempts）→ 如果修复成功则进入 runtime，否则 fail
- **行为改善** ✅

#### 潜在风险与缓解

**风险 1: LLM repair 可能引入新错误**

- LLM 可能误解错误信息，生成更糟的 IR
- **缓解**: governor.can_repair_v2 已有进度跟踪和重复检测，会停止无效修复

**风险 2: 增加响应时间**

- 每次 LLM repair 调用增加 ~10-30 秒
- max 3 attempts 可能增加 30-90 秒
- **缓解**: 这是可接受的代价，因为替代方案是直接 fail

**风险 3: LLM repair 可能违反 "不针对特定零件" 原则**

- 如果 repair prompt 包含零件特定信息，可能违反
- **缓解**: 使用通用的 repair prompt，只包含错误信息和原始 IR

#### 结论

**P1-2 是通用修复**，使用已有的 repair 框架，不影响合法输出。但需要谨慎设计 repair prompt，确保不包含零件特定信息。

## 五、关键约束与禁忌

### 5.1 绝对禁忌

以下做法**绝对禁止**，因为它们违反通用性原则：

1. ❌ **禁止在代码中判断零件类型**
   - 例如: `if "turbine" in prompt: special_handling()`
   - 这是特殊补丁，不是通用修复

2. ❌ **禁止在 prompt 中提及特定零件名称**
   - 例如: "如果是涡轮盘，应该..."
   - 这会引导 LLM 对特定零件特殊处理

3. ❌ **禁止为涡轮盘新增专用 op**
   - 例如: `cut_fir_tree_slot` op
   - 这违反了 "通用化、原子化、CAD化" 原则

4. ❌ **禁止修改验证规则以容忍非法输出**
   - 例如: 放宽 structure 验证，允许 root_node 不输出 body:solid
   - 这会破坏系统的 fail-closed 原则

5. ❌ **禁止单独处理 close_profile**
   - 例如: `if op == "close_profile": special_check()`
   - 应该基于 spec input_types 通用判断

### 5.2 必须遵循的原则

1. ✅ **修复必须基于通用规则**
   - 基于 op spec 的 input_types/output_types
   - 基于 count 阈值（与已有策略一致）

2. ✅ **修复必须与已有验证规则一致**
   - _fix_root_node 的修复与 structure 验证规则一致
   - L2 prompt 的规则与 typecheck 验证规则一致

3. ✅ **修复必须经过回归测试**
   - 所有现有测试必须通过
   - 所有合法零件类型必须验证

4. ✅ **修复必须有文档记录**
   - 记录修复的通用性分析
   - 记录影响范围
   - 记录回归测试结果

## 六、总结

### 6.1 推荐实施路径

**最小化、最安全的修复路径**:

1. **立即实施 P0-1 + P0-3**（~15 行代码 + ~5 行 prompt）
   - 解决本次测试 structure 失败
   - 完全通用，不影响任何合法零件
   - 风险极低

2. **验证后实施 P1-1**（~100 行代码）
   - 解决 geometry_postcheck 失败
   - 与已有 cut_rim_slot_pattern 策略一致
   - 需要验证 CadQuery compound tool 支持

3. **可选实施 P1-2**（~80 行代码）
   - 提高系统鲁棒性
   - 使用已有 repair 框架
   - 不影响合法输出

4. **暂缓 P0-2**
   - 仅在 P0-1 + P0-3 不足时实施
   - 实现复杂度高，风险中等

### 6.2 修复的通用性声明

**所有推荐修复（P0-1, P0-3, P1-1, P1-2）都是通用的**:

- 不包含任何零件类型判断
- 不包含任何零件特定逻辑
- 与已有验证规则一致
- 不影响任何合法 LLM 输出
- 经过合法零件类型回归测试验证

这些修复将使系统对所有零件都更健壮，包括但不限于：涡轮盘、齿轮、轴承、壳体、法兰、支架、板件、装配体等。

### 6.3 风险控制

**实施过程中的风险控制措施**:

1. 每个修复独立实施，独立测试
2. 每次修复后运行全量回归测试
3. 如果回归测试失败，立即回滚
4. 如果 P0-1 + P0-3 解决了问题，不实施 P0-2
5. 如果 P1-1 验证 CadQuery 不支持 compound tool，重新设计方案
