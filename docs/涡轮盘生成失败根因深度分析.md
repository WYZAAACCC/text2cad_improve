## 二、本次测试失败的根本原因

### 2.1 直接原因：LLM 创建 hallucinated passthrough 节点

LLM 在 `llm_raw.json` 中为 `disc_body` 和 `fir_tree_cutter` 两个组件各创建了一个 **passthrough 节点**作为"完成标记"：

**disc_body 组件的节点链**:
```
n_sketch_disc → n_disc_poly → n_disc_close → n_disc_revolve → n_disc_done (passthrough!)
```

**n_disc_done 节点的 raw 定义** (llm_raw.json:186-209):
```json
{
  "id": "n_disc_done",
  "op": "close_profile",           // ← close_profile 的 spec 输出是 profile
  "inputs": [{"node": "n_disc_revolve", "output": "body"}],  // ← 输入是 solid
  "outputs": [{"name": "body", "type": "solid"}]  // ← LLM 伪造！spec 应为 profile
}
```

**问题**: LLM 把 `close_profile`（一个修改 profile 的 op）误用为 passthrough 节点，输入 solid、输出 solid，作为"component 完成"的标记。但 `close_profile` 的 OperationSpec ([sketch_profile/dialect.py:126-132](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/sketch_profile/dialect.py#L126-L132)) 明确定义 `input_types=["profile"], output_types=["profile"]`。

### 2.2 autofix 修复链条断裂 — 三层失败

autofix 按顺序执行 27 个 fix（[auto_fixer.py:270-296](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L270-L296)），其中三个关键 fix 未能协同修复此问题：

#### 第 1 层: `fix_output_names` (auto_fixer.py:278) — 正确执行但引发后续问题

此 fix 根据 OperationSpec 修正输出类型。对 `n_disc_done`：
- **修改前** (LLM 伪造): `outputs=[{"name":"body","type":"solid"}]`
- **修改后** (按 close_profile spec): `outputs=[{"name":"profile","type":"profile"}]`

验证: `raw_fixed.json:222-227` 确认 outputs 已被改为 `[{"name":"profile","type":"profile"}]`。

#### 第 2 层: `fix_unknown_ops` (auto_fixer.py:286) — 不删除合法 op 节点

此 fix ([auto_fixer.py:614-677](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L614-L677)) 只删除 dialect registry 中不存在的 op（如 `cleanup`, `compose`）。`close_profile` 是 sketch_profile 的合法 op，所以 **`n_disc_done` 节点被保留**。

#### 第 3 层: `fix_root_node` (auto_fixer.py:289) — **关键 Bug**

此 fix ([auto_fixer.py:573-611](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L573-L611)) 的逻辑：

```python
def _fix_root_node(doc: dict) -> dict:
    ...
    for comp in doc.get("components", []):
        rn = comp.get("root_node", "")
        if rn and rn in node_ids:
            root_node_obj = next(...)
            if root_node_obj and root_node_obj.get("required", True):
                continue  # ← BUG: 当前 root_node 是 required=True，直接保留
        # 重新选择 root_node 的逻辑（仅 required=False 时触发）
        ...
```

**Bug**: 当 root_node 指向的节点是 `required=True` 时，立即 `continue` 保留，**不检查该节点是否真的输出 `body:solid`**。即使 `fix_output_names` 已经把 `n_disc_done` 的 outputs 改成了 `profile`（不再是 body:solid），`fix_root_node` 也不会重新选择正确的 body:solid 节点（如 `n_disc_revolve`）。

### 3.2 可能原因分析

#### 原因 A: 60 个 cutter union 导致 OCCT 拓扑损坏

[composition/handlers.py:179-222](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/composition/handlers.py#L179-L222) 的 `handle_circular_pattern_component`:

```python
result = body.translate((radius, 0, 0))
for i in range(1, count):  # count=60
    angle = start_angle + i * 360.0 / count
    placed = body.translate((radius, 0, 0)).rotate((0, 0, 0), (0, 0, 1), angle)
    result = result.union(placed)  # ← 60 次 union
```

60 个枞树槽 cutter 的 union 会产生极其复杂的 compound。OCCT 的 boolean 操作在大量重复 union 后可能出现拓扑不一致，导致 `Closed()` 返回 False。这是已知的 OCCT 限制 — 之前的修复（count > 6 时改用 sequential cut）正是为了解决类似问题，但 `circular_pattern_component` 仍然使用 union 策略。

#### 原因 B: boolean_cut 后几何未闭合

[composition/handlers.py:359-388](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/composition/handlers.py#L359-L388) 的 `handle_boolean_cut`:

```python
try:
    result = target.cut(tool)
except Exception:
    try:
        result = target.cut(tool, ctx.tolerance.boolean_fallback_tolerance)
    except Exception:
        raise RuntimeError(...)
return {"body": _store_solid(node, ctx, result)}
```

如果 `target.cut(tool)` 部分成功（如 cutter 与 target 不完全相交），可能返回未正确闭合的 solid。CadQuery 的 cut 操作在某些情况下会返回非闭合 solid，特别是当 cutter 几何复杂（枞树槽多级齿形）时。

#### 原因 C: cutter 几何本身有问题

LLM 创建的 fir-tree cutter 是 2D profile + extrude。但：
- `fillet_sketch` 在 CadQuery 2.7 中不可用（[handlers.py:956-959](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/sketch_profile/handlers.py#L956-L959) 已做 graceful degradation），意味着没有圆角
- 没有圆角的枞树槽有尖锐转角，可能导致 boolean_cut 后拓扑损坏
- LLM 创建的 profile 可能有自相交（17 个点的复杂多边形）

#### 原因 D: place_component + circular_pattern 双重定位

进度文档提到 LLM 可能先 `place_component` 再 `circular_pattern`，导致双重定位。L2 prompt 已添加禁止规则（prompts.py:146-148），但 LLM 可能不遵循。如果发生双重定位，cutter 会被放置到错误位置，boolean_cut 可能产生空结果或损坏几何。

## 四、架构层面缺陷

### 4.1 缺陷 A: 没有 LLM repair loop 在 L2 阶段

[main.py:430-447](file:///e:/auto_detection_process/app/text-to-cad/server/main.py#L430-L447) 的 `_run_pipeline` 在 L2 validation 失败后：

```python
canonical, report, bundle = validate_and_canonicalize_with_bundle(raw)
if not report.ok:
    try:
        fixed, af = auto_fix_with_report(raw, reg)
        if af.applied:
            canonical, report, bundle = validate_and_canonicalize_with_bundle(fixed)
    except Exception:
        pass
errs = [i for i in report.issues if i.severity=="error"]
if canonical is None or errs:
    _update_task(task_id, status="failed", progress=0,
                 error=f"Validation: {len(errs)} errors")
    return  # ← 直接失败，不调用 LLM repair
```

系统中存在 `RepairPatchV2` 和 `governor.can_repair_v2`（max 3 attempts），但 `_run_pipeline` **没有使用** LLM repair loop。如果 autofix 无法修复（如本例），直接 fail。

### 4.2 缺陷 B: L2 prompt 缺少禁止 passthrough 节点的规则

[prompts.py:72-171](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/skills/prompts.py#L72-L171) 的 `LEVEL2_AUTHORING_SYSTEM_PROMPT`:

- Rule 28: "The final root node must output 'body' of type 'solid'."
- Rule 23: "Every node output type must match OperationSpec output_types."

但**缺少**:
- "不要创建仅作为'完成标记'的 passthrough 节点"
- "root_node 必须直接指向产生 solid 的节点（如 extrude_profile, revolve_profile, boolean_cut），不能指向 close_profile 等修改 profile 的节点"
- "每个组件的最后一个节点必须是产生 solid 的节点，不要在后面追加额外的标记节点"

### 4.3 缺陷 C: canonicalize 不验证 output type 与 spec 的匹配

[canonicalize.py:121-126](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/validation/canonicalize.py#L121-L126) 将 raw outputs **原封不动**复制到 canonical：

```python
resolved_outputs = []
for decl in node.outputs:
    resolved_outputs.append(CanonicalValueDecl(
        name=decl.name, type=decl.type,
        value_id=f"{decl.type}:{node.component}:{node.id}:{decl.name}",
    ))
```

不验证 `decl.type` 是否与 OperationSpec 的 `output_types` 匹配。只有 typecheck 阶段会验证，但 typecheck 在 structure 之后。

### 4.4 缺陷 D: fix_unknown_ops 不检测 op 误用

[auto_fixer.py:614-677](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L614-L677) 的 `_fix_unknown_ops` 只检查 op 是否在 dialect registry 中存在，**不检查输入输出类型是否匹配 spec**。`close_profile` 是合法 op，所以被误用为 passthrough 时不会被删除。

### 4.5 缺陷 E: _fix_root_node 不验证输出类型

[auto_fixer.py:586-611](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/authoring/auto_fixer.py#L586-L611) 的 `_fix_root_node`:

```python
if root_node_obj and root_node_obj.get("required", True):
    continue  # 当前 root_node 是 required=True，保留
```

**只检查 required=True，不检查输出类型**。即使 `fix_output_names` 已经把 outputs 改成了非 body:solid，`fix_root_node` 也不会重新选择。

## 五、修复方案

### 5.1 P0 修复 — 立即修复（解决本次测试失败）

#### P0-1: 修复 `_fix_root_node` 验证输出类型

**文件**: `auto_fixer.py:573-611`

**修改**: 在 `required=True` 检查后，增加 `body:solid` 输出验证。如果当前 root_node 不输出 body:solid，重新选择。

```python
def _fix_root_node(doc: dict) -> dict:
    ...
    for comp in doc.get("components", []):
        rn = comp.get("root_node", "")
        if rn and rn in node_ids:
            root_node_obj = next(...)
            if root_node_obj and root_node_obj.get("required", True):
                # ← 新增：验证 root_node 是否真的输出 body:solid
                has_body_solid = any(
                    o.get("name") == "body" and o.get("type") == "solid"
                    for o in root_node_obj.get("outputs", [])
                )
                if has_body_solid:
                    continue  # 真正有效的 root_node，保留
                # fall through 重新选择
        # 重新选择 root_node 的逻辑
        ...
```

**预期效果**: `n_disc_done` (outputs=profile) 会被识别为无效 root_node，重新选择 `n_disc_revolve` (outputs=body:solid) 作为 root_node。

#### P0-2: 增加 passthrough 节点检测与清理

**文件**: `auto_fixer.py` 新增 `_strip_passthrough_nodes` 函数

**逻辑**: 检测 op 的 input_types 和 output_types，如果节点输入 solid 输出 solid 但 op 的 spec 不允许（如 close_profile spec 是 profile→profile），则删除该节点并修复引用。

```python
def _strip_passthrough_nodes(doc: dict, dialect_registry=None) -> dict:
    """删除 LLM 误用的 passthrough 节点。
    
    检测：节点的实际输入类型与 op spec 的 input_types 不匹配时，删除该节点。
    例如 close_profile 接收 solid 输入（spec 应为 profile），是 passthrough 误用。
    """
    # 1. 构建节点 id → 实际输入类型映射
    # 2. 对每个节点，检查其输入类型是否匹配 op spec 的 input_types
    # 3. 不匹配的节点删除，并将下游引用重定向到其输入源
    ...
```

#### P0-3: 在 L2 prompt 中添加禁止 passthrough 节点的规则

**文件**: `prompts.py` 的 `LEVEL2_AUTHORING_SYSTEM_PROMPT`

**新增规则**:
```
34. Do NOT create passthrough/marker nodes at the end of a component.
    The root_node must directly point to a solid-producing node
    (extrude_profile, revolve_profile, boolean_cut, boolean_union, etc.).
    Do NOT add close_profile or other profile-modifying nodes after a solid-producing node.
35. close_profile may ONLY follow add_polyline/add_line_segment/add_arc_segment/add_circle.
    close_profile MUST NOT follow extrude_profile, revolve_profile, or any solid-producing op.
```

### 5.2 P1 修复 — 短期修复（解决 geometry_postcheck 失败）

## 八、结论

### 8.1 本次测试失败的根因链

```
LLM 误解 "每个 component 必须有 root_node"
  → 创建 passthrough 节点 (close_profile 误用为 solid→solid)
  → 伪造 outputs=[body:solid]

fix_output_names 修正 outputs 为 [profile:profile]
  → 但 fix_root_node 不检查输出类型，保留无效 root_node

structure 验证检查 root_node 必须输出 body:solid
  → n_disc_done outputs=[profile:profile] → 失败

没有 LLM repair loop
  → 直接 fail，无法修复
```

### 8.2 核心修复优先级

| 优先级 | 修复项 | 解决问题 | 工作量 |
|--------|--------|----------|--------|
| **P0-1** | 修复 `_fix_root_node` 验证输出类型 | 本次测试 structure 失败 | 小（~10 行代码） |
| **P0-2** | 增加 passthrough 节点检测与清理 | 防止类似问题 | 中（~50 行代码） |
| **P0-3** | L2 prompt 添加禁止 passthrough 规则 | 从源头防止 LLM 创建 passthrough | 小（~5 行 prompt） |
| **P1-1** | circular_pattern 使用 sequential cut | geometry_postcheck 失败 | 中（~100 行代码） |
| **P1-2** | 引入 LLM repair loop | 提高鲁棒性 | 中（~80 行代码） |

### 8.3 建议立即实施 P0-1 + P0-3

P0-1（修复 `_fix_root_node`）和 P0-3（L2 prompt 规则）是最低成本、最高收益的修复：

1. **P0-1** 直接解决本次测试失败：`_fix_root_node` 会重新选择 `n_disc_revolve` 作为 root_node
2. **P0-3** 从源头防止 LLM 创建 passthrough 节点
3. 两者结合后，即使 LLM 仍然创建 passthrough 节点，autofix 也能正确修复 root_node 指向

实施 P0-1 + P0-3 后，本次测试的 IR 应该能通过 structure 验证，进入 runtime 阶段。如果 runtime 阶段的 geometry_postcheck 仍然失败（问题 #2），则需要进一步实施 P1-1（sequential cut 策略）。
