# 榫槽切割策略鲁棒性优化计划

## Summary

评估并优化 `handle_cut_rim_slot_pattern` 的切割策略，确保修复具有通用性和正向性，不是"为涡轮盘打的补丁"。将当前的无条件 sequential cut 升级为 **count 阈值切换策略**，在小 count 场景（皮带轮、联轴器等）保留高性能的 union-then-cut，在大 count 场景（涡轮盘）使用稳健的 sequential cut，与已有的 `handle_cut_circular_hole_pattern` 策略对齐。

---

## Current State Analysis

### 当前修复（无条件 sequential cut）

已将 `handle_cut_rim_slot_pattern` 的"union-then-cut"策略改为"sequential cut"（[handlers.py:259-292](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/handlers.py#L259-L292)）。

### 鲁棒性评估

| 维度 | 评估 | 说明 |
|---|---|---|
| 通用性 | ✅ | sequential cut 是通用 boolean 策略，适用于所有 rim slot pattern |
| 正确性 | ✅ | 避免 union 拓扑累积，符合 OCCT 内核限制 |
| 稳健性 | ✅ | 大 count 场景避免崩溃，单槽失败时跳过 |
| 性能 | ⚠️ | 小 count（≤6）场景略逊于 union-then-cut，但差异可忽略 |
| 一致性 | ⚠️ | 与 `handle_cut_circular_hole_pattern` 的条件切换策略不一致 |

### 是否是"为涡轮盘打的补丁"？

**不是补丁，是正向修复**，但可以进一步优化：

1. **通用性证据**：`cut_rim_slot_pattern` 的 schema `count` 范围是 2-360（[models.py:142-145](file:///e:/auto_detection_process/integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/bases/axisymmetric/models.py#L142-L145)），典型零件包括涡轮盘（40-80）、皮带轮、联轴器等。sequential cut 对所有 count 都能产生正确结果。
2. **非补丁证据**：`handle_cut_circular_hole_pattern`（[handlers.py:166](file:///e:/auto_detection_process/integrators/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/handlers.py#L166)）在 `count > 6` 时已采用相同的 sequential cut 策略，这是项目已有的通用模式。
3. **性能影响**：对小 count（如 4-6），union-then-cut 约 5 次 union + 1 次 cut；sequential cut 约 6 次 cut。boolean 操作总数相同，但 sequential cut 每次 cut 的 result body 逐渐复杂，总体略慢。差异在毫秒级，可忽略。

### count 典型值范围（来自探索）

| 来源 | count | 说明 |
|---|---|---|
| schema 硬约束 | 2-360 | `ge=2, le=360` |

### 关键发现：策略不一致

`handle_cut_circular_hole_pattern`（[handlers.py:166-184](file:///e:/auto_detection_process/integrators/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/handlers.py#L166-L184)）已实现条件切换：
- `is_narrow_body or count > 6` → sequential cut（逐个 cut，容错跳过）
- 否则 → `polarArray` + single cut（性能最优）

但 `handle_cut_rim_slot_pattern` 当前是无条件 sequential cut，未对齐此模式。

## Proposed Changes

### Change 1: 添加 count 阈值切换（核心改进）

将无条件 sequential cut 改为条件切换，与 `handle_cut_circular_hole_pattern` 的策略对齐。

**文件**: `integrations/engineering_tools/src/seekflow_engineering_tools/generative_cad/dialects/axisymmetric/handlers.py`

**修改位置**: `handle_cut_rim_slot_pattern` 函数（第 257-298 行）

**策略**:
- `count <= 6`: union-then-cut（原策略，对小 count 性能最优）
- `count > 6`: sequential cut（新策略，对大 count 稳健性最优）

**阈值选择 6 的理由**:
1. 与 `handle_cut_circular_hole_pattern`（第 166 行 `count > 6`）完全一致，保持代码风格统一
2. 6 个 fir-tree 槽的 union 约 48 条边（6×8），远低于 OCCT 崩溃阈值
3. 6 是常见小零件（皮带轮、联轴器、法兰）的典型槽数上限
4. count=12 的单元测试会走 sequential cut 分支，验证大 count 路径

**修改后的代码逻辑**:

```python
base_cutter = wp.close().extrude(bb.zlen + 10, both=True)

if count <= 6:
    # 小 count: union-then-cut 策略（性能最优）
    # 6 个槽的 union 约 48 条边，远低于 OCCT 崩溃阈值
    combined = base_cutter
    for i in range(1, count):
        angle = math.degrees(i * 2 * math.pi / count)
        combined = combined.union(base_cutter.rotate((0, 0, 0), (0, 0, 1), angle))
    result = body.cut(combined)
else:
    # 大 count: sequential cut 策略（稳健性最优）
    # 避免 union 拓扑累积导致 OCP 崩溃（60 槽 union 需处理 ~480 条交线）
    # 与 handle_cut_circular_hole_pattern (count > 6) 的稳健策略一致
    result = body
    success_count = 0
    for i in range(count):
        angle = math.degrees(i * 2 * math.pi / count)
        try:
            cutter = base_cutter.rotate((0, 0, 0), (0, 0, 1), angle)
            result = result.cut(cutter)
            success_count += 1
        except Exception as slot_e:
            ctx.warnings.append(
                f"cut_rim_slot_pattern on '{node.id}': slot {i} at "
                f"angle {angle:.1f}° failed, skipping. Error: {slot_e}"
            )

    if success_count == 0:
        return handle_feature_failure(
            node=node, ctx=ctx, original_body=body,
            op_name="cut_rim_slot_pattern",
            reason=f"all {count} slots failed to cut",
        )
    if success_count < count:
        ctx.warnings.append(
            f"cut_rim_slot_pattern on '{node.id}': {count - success_count} of "
            f"{count} slots skipped due to OCCT errors."
        )
        ctx.degraded_features.append({
            "node_id": node.id, "op": "cut_rim_slot_pattern",
            "reason": f"{count - success_count}/{count} slots skipped",
        })
```

**为什么这个修复是正向的、通用的**：

1. **不是涡轮盘补丁**：阈值切换是通用策略，对任何 count 都选择最优路径
2. **不降低其他模型性能**：小 count（皮带轮等）走 union-then-cut，性能不变
3. **提升大 count 稳健性**：大 count（涡轮盘）走 sequential cut，避免崩溃
4. **与项目已有模式一致**：对齐 `handle_cut_circular_hole_pattern` 的条件切换模式
5. **保留容错能力**：大 count 路径有单槽失败跳过机制

## Assumptions & Decisions

1. **阈值选择 6**：与 `handle_cut_circular_hole_pattern` 完全一致，保持代码风格统一
2. **6 个 fir-tree 槽的 union 安全**：约 48 条边，远低于 OCCT 崩溃阈值（~480 条边才崩溃）
3. **不修改 schema 上限（360）**：超出本次范围，且 sequential cut 已能处理大 count
4. **不添加 preflight 检查**：超出本次范围，handler 层的容错已足够
5. **不修复 planner 规则不对称**：`cut_rim_slot_pattern` 缺少 `large_pattern_risk`（count>=120）警告，但这是 planner 层的问题，不影响 handler 的鲁棒性
