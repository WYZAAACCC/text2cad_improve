# 架构升级审查问题修复计划

## Summary

上一轮架构升级修复完成后，深度审查发现 **3 个遗留问题**，导致 fail-closed 策略在某些场景下被绕过：

| 问题 | 严重度 | 位置 | 根因 |
|---|---|---|---|
| A | 严重 | `bases/axisymmetric/preflight.py:81-102` | Z 重叠检测在错误的文件中，是死代码 |
| B | 中等 | `dialects/axisymmetric/handlers.py:60-69` | "区域描述"检测条件过严，不会触发实际场景 |
| C | 中等 | `authoring/auto_fixer.py:711` | 同问题 B，检测条件过严 |

**核心风险**：如果 LLM 修复了 A005 共面问题（移除 r=60 站点）但保留区域描述风格（不同 Z 区间），系统会 silently 产生自相交多边形，导致 revolve 失败或错误几何。

### 边界条件验证

Z 重叠检测条件 `zf1 < zr2 and zf2 < zr1`（严格小于）对合法外轮廓连续段**不会误报**：

**TURBINE_DISC_REFERENCE_EXAMPLE**（合法）：
- (0,5), (5,10), (10,30), (30,35), (35,40)
- 相邻段 Z 边界相等但不严格小于 → 不重叠 ✓

**WASHER_EXAMPLE**（合法）：
- (0,2), (2,12), (12,13)
- 同上，不重叠 ✓

**LLM 区域描述输出**（非法）：
- (5,6), (6,115), (5,42.5), (42.5,77.5), (42.5,77.5), (20,100)
- 多对 Z 区间严格重叠 → 触发检测 ✓

## Proposed Changes

### 修复 1：将 Z 重叠检测移到 dialect.py 的 preflight_component（问题 A）

**文件**：`dialects/axisymmetric/dialect.py`

**位置**：`preflight_component` 方法，A001 检测之后、A005 之前（约第 145 行之后）

**添加内容**：
```python
# ── A006: revolve_profile Z 重叠检测 ──
# 检测"区域描述"风格：多个 station 的 Z 区间重叠
# 这是 LLM 把 bore/hub/web/rim 分区域输出的典型模式，违反 Z 单值约束
for n in nodes:
    if n.op != "revolve_profile":
        continue
    ps = n.typed_params.get("profile_stations") or n.params.get("profile_stations", [])
    if len(ps) < 2:
        continue
    z_ranges = [(s.get("z_front_mm", 0), s.get("z_rear_mm", 0)) for s in ps]
    overlap_count = 0
    for i, (zf1, zr1) in enumerate(z_ranges):
        for j, (zf2, zr2) in enumerate(z_ranges[i+1:], i+1):
            if zf1 < zr2 and zf2 < zr1:  # Z 区间严格重叠
                overlap_count += 1
    if overlap_count > 0:
        issues.append(ValidationIssue(
            stage=stage, code="a006_profile_z_overlap",
            message=(
                f"revolve_profile stations have {overlap_count} overlapping Z ranges. "
                f"This indicates regional description style (bore/hub/web/rim), "
                f"which violates the Z single-valued outer contour constraint. "
                f"Convert to outer contour continuous segments where each Z position "
                f"has exactly one radius."
            ),
            severity="error", node_id=n.id,
        ))
```

**why**：将 Z 重叠检测放到正确的 validation pipeline 调用路径上，使其在 v0.2 系统中实际执行。

# 这是 LLM 把 bore/hub/web/rim 分区域输出的典型模式，会导致 sort-by-Z
# 算法 silently 产生错误形状。必须 fail-closed 报错。
if len(stations) >= 2:
    z_ranges = [(s.get("z_front_mm", 0), s.get("z_rear_mm", 0)) for s in stations]
    overlap_count = 0

## Assumptions & Decisions

### 假设
1. v0.2 dialect 系统是当前生产路径（基于测试诊断报告确认）
2. `bases/axisymmetric/runner.py`（v0.1 legacy）不在 v0.2 流程中使用
3. Z 重叠检测条件 `zf1 < zr2 and zf2 < zr1`（严格小于）对合法外轮廓不会误报（已验证）

### 决策
1. **删除 vs 保留 `bases/axisymmetric/preflight.py` 的死代码**：删除。这是上一轮添加的错误代码，属于"自己的 mess"。
2. **handler 检测 vs preflight 检测的重复**：保留两者。preflight 是第一道防线（validation 阶段），handler 是最后防线（runtime 阶段）。即使 preflight 漏检，handler 也会 fail-closed。
3. **autofix 策略**：保持"标记 _autofix_hints 并跳过"，不直接修改。区域描述风格无法安全自动转换为连续段，需要 LLM repair。
